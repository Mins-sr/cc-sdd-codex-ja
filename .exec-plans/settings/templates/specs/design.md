# 設計ドキュメントテンプレート

---
**目的**: 実装者が異なっても実装の一貫性を確保し、解釈のブレを防ぐのに十分な詳細を提供する。

**アプローチ**:
- 実装の決定に直接関係する必要なセクションを含める
- 実装エラーを防ぐために不可欠でない限り、任意のセクションは省略する
- 詳細度を機能の複雑さに合わせる
- 長い文章よりも図や表を優先する

**警告**: 1000行に近づく場合は、機能が過度に複雑であることを示しており、設計の簡素化や複数の仕様への分割が必要になる可能性があります。
---

> セクションの順序は、明確さを向上させる場合に並べ替えることができます（例: 「要件のトレーサビリティ」を前の方に持ってくる、または「データモデル」を「アーキテクチャ」に近づけるなど）。各セクション内では、レビューアが一貫して目を通せるように **要約 → スコープ → 決定事項 → 影響/リスク** の流れを維持してください。

## 境界のコミットメント

この仕様の責任範囲を具体的な言葉で述べます。これをアーキテクチャ、タスク、および後のバリデーションのアンカー（基準）として扱います。

### この仕様が所有するもの
- この仕様が責任を負う機能と振る舞い
- この仕様が所有する、または権威を持つデータ
- この仕様が定義、または安定化させるインターフェースやコントラクト

### 境界外
- この仕様が明示的に所有しない関連する関心事
- 他の仕様、既存のサブシステム、または後のフェーズに延期される作業
- この仕様が「もう一つの項目」として吸収してはならない変更

### 許可された依存関係
- この設計が依存する可能性のある上流のシステム/仕様/コンポーネント
- この設計が使用する可能性のある共有インフラストラクチャ
- 違反してはならない依存関係の制約

### 再バリデーションのトリガー
依存する仕様や消費者に統合の再チェックを強制すべき変更の種類をリストアップします。

- コントラクトの形状の変更
- データ所有権の変更
- 依存関係の方向の変更
- 起動/ランタイムの前提条件の変更

## アーキテクチャ

> 詳細な調査ノートは背景としてのみ `research.md` を参照してください。設計上のすべての決定とコントラクトをここに捉え、レビューアにとって `design.md` が自己完結型になるように保ちます。
> テキストに重要な決定事項を捉え、構造的な詳細は図解に持たせます — 文章で同じ情報を繰り返さないでください。
> 以下のサポートセクションは、責任の境界、依存関係のルール、または統合の継ぎ目を実質的に明確にする場合を除き、可能な限り軽く保つべきです。

### 既存アーキテクチャの分析 (該当する場合)
既存のシステムを変更する場合:
- 現在のアーキテクチャパターンと制約
- 尊重すべき既存のドメイン境界
- 維持しなければならない統合ポイント
- 対処、または回避された技術的負債

### アーキテクチャパターンと境界マップ
**推奨**: 選択したアーキテクチャパターンとシステムの境界を示す Mermaid の図を含めます（複雑な機能には必須、単純な追加には任意）。

## ファイル構造の計画

この機能のディレクトリ構造とファイルの責任をマッピングします。このセクションは、タスクの `_Boundary:_` アノテーションと実装のタスクブリーフを直接駆動します。適切な詳細レベルを使用してください:

- **小さな機能**: 責任を持たせた個々のファイルをリスト化する
- **大きな機能**: ディレクトリレベルの構造 + ドメイン/モジュールごとのパターンを説明し、自明でないファイルのみを個別にリスト化する

### ディレクトリ構造
```
src/
├── domain-a/              # Domain A の責任範囲
│   ├── controller.ts      # エンドポイントハンドラ
│   ├── service.ts         # ビジネスロジック
│   └── types.ts           # ドメインの型定義
├── domain-b/              # Domain B (domain-a と同じパターン)
└── shared/
    └── cross-cutting.ts   # 自明でないもの: なぜこれが存在するか
```

> 繰り返される構造については、パターンを一度だけ説明します（例:「domain-b は domain-a と同じパターンに従う」）。パスから責任が明白でない場合にのみ、個別のファイルをリスト化してください。

### 変更されるファイル
- `path/to/existing.ts` — 何がなぜ変わるのか

> 各ファイルは1つの明確な責任を持つべきです。一緒に変更されるファイルをグループ化します。繰り返される構造については、各ファイルをリスト化するのではなく、パターンを一度説明します。
> コンポーネントとインターフェースのセクションですでに説明されている内容を重複させないでください — コンポーネントがマッピングされる物理的なファイルの配置に焦点を当てます。

## システムフロー

複雑（非自明）なフローを説明するために必要な図のみを提供します。純粋な Mermaid 構文を使用してください。一般的なパターン:
- シーケンス（複数の関係者間の相互作用）
- プロセス / 状態（分岐ロジックやライフサイクル）
- データ / イベントフロー（パイプライン、非同期メッセージング）

単純な CRUD の変更の場合は、このセクションを完全にスキップしてください。
> 各ステップを繰り返すのではなく、図の後にフローレベルの決定（例: ゲート条件、再試行）について簡潔に説明してください。

## コンポーネントとインターフェース

コンポーネントごとの詳細に掘り下げる前に、クイックリファレンスを提供します。

- 要約は表またはコンパクトなリストで行うことができます。表の例:
  | コンポーネント | ドメイン/レイヤー | 意図 | 要件カバー | 主な依存関係 (P0/P1) | コントラクト |
  |-----------|--------------|--------|--------------|--------------------------|-----------|
  | ExampleComponent | UI | XYZを表示 | 1, 2 | GameProvider (P0), MapPanel (P1) | Service, State |
- 新しい境界を導入するコンポーネント（例: ロジックフック、外部統合、永続化）のみが完全な詳細ブロックを必要とします。単純なプレゼンテーションコンポーネントは、要約の行と短い実装メモだけで十分です。

## データモデル

この機能に伴って変化するデータの領域に焦点を当てます。

### ドメインモデル

### 論理データモデル

## エラー処理

### エラー戦略
各エラータイプに対する具体的なエラー処理のパターンとリカバリメカニズム。

### エラーのカテゴリとレスポンス
**ユーザーエラー** (4xx): 無効な入力 → フィールドレベルのバリデーション; 未承認 → 認証の案内; 見つからない → ナビゲーションのヘルプ
**システムエラー** (5xx): インフラ障害 → グレースフルデグラデーション; タイムアウト → サーキットブレーカー; 枯渇 → レート制限  
**ビジネスロジックエラー** (422): ルール違反 → 条件の説明; 状態の競合 → 移行の案内

**プロセスフローの視覚化** (複雑なビジネスロジックが存在する場合):
ビジネスワークフローを伴う複雑なエラーシナリオの場合にのみ、Mermaid のフローチャートを含めます。

### 監視
エラートラッキング、ログ記録、およびヘルス監視の実装。

## テスト戦略

### デフォルトセクション (ドメインに合わせて名前/セクションを適応させてください)
- ユニットテスト: コアとなる機能/モジュールから 3〜5 項目 (例: 認証メソッド、サブスクリプションロジック)
- 統合テスト: 複数コンポーネントをまたぐフローの中から 3〜5 項目 (例: Webhook処理、通知)
- E2E/UI テスト (該当する場合): クリティカルなユーザーパスから 3〜5 項目 (例: フォーム、ダッシュボード)
- パフォーマンス/負荷 (該当する場合): 3〜4 項目 (例: 並行処理、大量の操作)
