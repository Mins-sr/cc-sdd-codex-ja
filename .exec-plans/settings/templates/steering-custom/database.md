# データベースの標準

[目的: スキーマ設計、クエリ、マイグレーション、および整合性を導くこと]

## 哲学
- ドメインを優先してモデル化; 正確さのあとに最適化する
- 明示的な制約を好む; データベースに不変性を強制させる
- 必要なものだけをクエリする; 最適化する前に測定する

## 命名と型
- テーブル: `snake_case`, 複数形 (`users`, `order_items`)
- カラム: `snake_case` (`created_at`, `user_id`)
- 外部キー: `{table}.id` を参照する `{table}_id`
- 型: タイムゾーンを考慮したタイムスタンプ; 強力なID; 正確な金額用型

## リレーションシップ
- 1:N: 子テーブルに外部キー
- N:N: 複合キーを持つ結合テーブル
- 1:1: 外部キー + UNIQUE

## マイグレーション
- 不変のマイグレーション; 常にロールバックを追加する
- 小さく焦点の絞られたステップ; まずは非本番環境でテストする
- 命名: `{seq}_{action}_{object}` (例: `002_add_email_index`)

## クエリのパターン
- シンプルな CRUD と安全性には ORM を使用し、複雑でパフォーマンスが重要な場合は生の SQL を使用する
- N+1 を回避する（Eager Load / バッチング）; 大規模セットはページネーションする
- 外部キーと頻繁にフィルタリング/ソートされる列にインデックスを作成する

## 接続とトランザクション
- プーリングを使用する（ワークロードに基づいたサイズ/タイムアウト指定）
- 作業単位ごとに1つの接続; 迅速にクローズ/返却する
- マルチステップの変更はトランザクションでラップする

## データの整合性
- NOT NULL / UNIQUE / CHECK / 外部キー 制約を使用する
- 適切な場合は DB 側でバリデーションする（多層防御）
- 整合性のある派生値には生成列の使用を優先する

## バックアップとリカバリ
- 保持期間を設けた定期バックアップ; リストアをテストする
- RPO/RTO 目標を文書化する; バックアップジョブを監視する

---
_環境固有の設定ではなく、パターンと決定事項に焦点を当ててください。_
