# 認証と認可の標準

[目的: 認証モデル、トークン/セッションのライフサイクル、権限チェック、およびセキュリティを統一する]

## 哲学
- 明確な分離: 認証（誰か）と認可（何を）
- デフォルトで安全: 最小権限、フェイルクローズド、短寿命トークン
- UXを意識: リスクが高い場所では摩擦を伴い、それ以外はスムーズに

## 認証

### 手法 (選択と根拠)
- 選択肢: JWT, Session, OAuth2, ハイブリッド
- 選択: [私たちの手法] 理由: [理由]

### フロー (上位レベル)
```
1) ユーザーが身元を証明する（資格情報またはプロバイダー）
2) サーバーが検証し、トークン/セッションを発行する
3) クライアントはリクエストごとにトークンを送信する
4) サーバーがトークンを検証して処理を進める
```

### トークン/セッションのライフサイクル
- 保存: httpOnly Cookie または Authorization ヘッダー
- 有効期限: 短命のアクセス、より長いリフレッシュ（使用する場合）
- リフレッシュ: トークンのローテーション; 失効を尊重する
- 失効: ログアウト時または漏洩時のブラックリスト化/ローテーション

### セキュリティパターン
- TLS を強制する; 回避可能な場合はトークンを JS に絶対に露出しない
- トークンをオーディエンス/発行者にバインドする; 最小限のクレームを含める
- 機密性の高い操作については、デバイスバインディングとIP/リスクチェックを検討する

## 認可

### 権限モデル
- いずれかを選択: RBAC / ABAC / 所有権ベース / ハイブリッド
- 役割/属性を一元的に定義する; コードベース全体でのハードコーディングを避ける

### チェック (どこで強制するか)
- ルート/ミドルウェア: 粗粒度のゲート
- ドメイン/サービス: 細粒度の決定
- UI: 条件付きレンダリング（セキュリティとして依存しない）

パターン例:
```typescript
requirePermission('resource:action'); // ルート
if (!user.can('resource:action')) throw ForbiddenError(); // ドメイン
```

### 所有権
- パターン: 所有者 または 特権を持つ役割が行動できる
- 変更を加える前にエンティティ層の境界で検証する

## パスワードと MFA
- パスワード: 強力なポリシー、ハッシュ化 (bcrypt/argon2)、絶対にプレーンテキストにしない
- リセット: 時間制限付きトークン、単一使用、ユーザーに通知する
- MFA: リスクの高い操作のためのステップアップ（ポリシードリブン）

## API 間認証
- API キーまたは OAuth クライアント資格情報を使用する
- キーのスコープは最小限にする; ローテーションと使用状況の監査を行う
- アイデンティティ（ユーザー/キー）でレート制限を行う
