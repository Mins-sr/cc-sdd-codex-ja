# テストの標準

[目的: 何をテストするか、テストをどこに配置するか、どのように構成するかを導くこと]

## 哲学
- 実装ではなく振る舞いをテストする
- 高速で信頼性の高いテストを優先する; 壊れやすいモックを最小限に抑える
- クリティカルパスは深くカバーする; 100%の追求よりも広さを重視する

## 構成
オプション:
- 同じ場所に配置: `component.tsx` + `component.test.tsx`
- 分離: `/src/...` と `/tests/...`
デフォルトを1つ選択し、理由がある場合は例外を許可します。

命名規則:
- ファイル: `*.test.*` または `*.spec.*`
- スイート: テスト対象; ケース: 期待される振る舞い

## テストの種類
- 単体（Unit）: 単一のユニット、モック化された依存関係、非常に高速
- 統合（Integration）: 複数のユニットの組み合わせ、外部システムのみをモック化
- E2E: 完全なフロー、モックは最小限、クリティカルな経路のみ

## 構造 (AAAパターン)
```typescript
it('Yの時にXを行う', () => {
  // Arrange (準備)
  const input = setup();
  // Act (実行)
  const result = act(input);
  // Assert (検証)
  expect(result).toEqual(expected);
});
```

## モックとデータ
- 外部システム（API/DB）をモック化する; テスト対象システムを決してモック化しない
- ファクトリ/フィクスチャを使用する; テスト間で状態をリセットする
- テストデータは最小限にし、意図がわかるように保つ

## カバレッジ
- 目標: [全体 %]; 重要なドメインではより高くする
- CI でしきい値を強制する; 例外にはレビューと合理的な理由が必要
