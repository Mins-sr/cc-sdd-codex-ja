# API 標準

[目的: 命名、構造、認証、バージョニング、およびエラーの一貫した API パターン]

## 哲学
- 予測可能でリソース指向な設計を好む
- コントラクトを明示的にし、破壊的変更を最小限に抑える
- デフォルトで安全（認証ファースト、最小権限）

## エンドポイントのパターン
```
/{version}/{resource}[/{id}][/{sub-resource}]
```
例:
- `/api/v1/users`
- `/api/v1/users/:id`
- `/api/v1/users/:id/posts`

HTTP 動詞:
- GET (読み取り、安全、べき等)
- POST (作成)
- PUT/PATCH (更新)
- DELETE (削除、べき等)

## リクエスト/レスポンス

リクエスト (典型的な形式):
```json
{ "data": { ... }, "metadata": { "requestId": "..." } }
```

成功:
```json
{ "data": { ... }, "meta": { "timestamp": "...", "version": "..." } }
```

エラー:
```json
{ "error": { "code": "ERROR_CODE", "message": "...", "field": "optional" } }
```
(ルールの詳細についてはエラー処理を参照してください。)

## ステータスコード (パターン)
- 2xx: 成功 (200 読み取り, 201 作成, 204 削除)
- 4xx: クライアントの問題 (400 バリデーション, 401/403 認証関係, 404 不足)
- 5xx: サーバーの問題 (500 一般的, 503 利用不可)
結果を最も反映するステータスを選択してください。

## 認証
- 標準的な場所に認証情報を配置
```
Authorization: Bearer {token}
```
- ビジネスロジックの前に未認証のリクエストを拒否する

## バージョニング
- URL/ヘッダー/メディアタイプを介してバージョン管理を行う
- 破壊的変更 → 新バージョン
- 非破壊的変更 → 同一バージョン
- 非推奨化期間と連絡手段を提供する

## ページネーション/フィルタリング (該当する場合)
- ページネーション: `page`, `pageSize` またはカーソルベース
- フィルタリング: 明示的なクエリパラメータ
- ソート: `sort=field:asc|desc`
ページネーションのメタデータは `meta` 内で返します。

---
_エンドポイントのカタログではなく、パターンと決定事項に焦点を当ててください。_
