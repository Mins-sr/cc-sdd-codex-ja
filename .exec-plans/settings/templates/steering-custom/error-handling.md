# エラー処理の標準

[目的: エラーの分類、形式、伝播、ログ記録、および監視の方法論を統一する]

## 哲学
- 可能な限り迅速に失敗させる; システムの境界では段階的に機能を低下させる（グレースフルデグラデーション）
- スタック全体で一貫したエラーの形式（人間および機械が読み取れるもの）
- 既知のエラーは発生源の近くで処理する; 未知のエラーはグローバルハンドラに表面化させる

## 分類 (発生源に応じた処理の決定)
- クライアント: 入力/バリデーション/ユーザーの操作による問題 → 4xx
- サーバー: システム障害/予期せぬ例外 → 5xx
- ビジネス: ルール/状態の違反 → 4xx (例: 409)
- 外部: サードパーティ/ネットワークの障害 → コンテキストに応じて 5xx または 4xx にマッピングする

## エラーの形式 (単一の標準フォーマット)
```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "人間が読めるメッセージ",
    "requestId": "trace-id",
    "timestamp": "ISO-8601"
  }
}
```
原則: 安定したコード列挙型 (enum)、シークレットを含めない、トレース情報を含める。

## 伝播 (どこで変換するか)
- API層: ドメインエラーを変換 → HTTPステータス + 標準ボディ
- サービス層: 型付けされたビジネスエラーをスローし、文字列ベースのエラーを避ける
- データ/外部層: プロバイダーのエラーを安全かつアクション可能なコードでラップする
- 未知のエラー: グローバルハンドラにバブルアップさせる → 500 + 一般的なメッセージ

パターン例:
```typescript
try { return await useCase(); }
catch (e) {
  if (e instanceof BusinessError) return respondMapped(e);
  logError(e); return respondInternal();
}
```

## ログ記録 (ノイズよりコンテキスト)
記録対象: 操作名、userId（利用可能な場合）、エラーコード、メッセージ、スタックトレース、requestId、最小限のコンテキスト。
記録禁止: パスワード、トークン、シークレット、完全な個人情報(PII)、機密データを含む完全なボディ。
レベル: ERROR (失敗)、WARN (回復可能/エッジケース)、INFO (主要イベント)、DEBUG (診断)。

## 再試行 (安全な場合のみ)
再試行する条件: ネットワーク/タイムアウト/一時的な 5xx かつ、操作がべき等である場合。
再試行しない条件: 4xx、ビジネスエラー、べき等でないフロー。
戦略: 指数バックオフ + ジッター、最大試行回数の上限設定; べき等性キーを必須とする。

## 監視とヘルスチェック
トラッキング対象: コード/カテゴリ別のエラー率、レイテンシ、飽和度; スパイクや SLI 違反でアラートを出す。
ヘルスの公開: `/health` (生存)、`/health/ready` (準備完了)。エラーをトレースにリンクさせる。
