# セキュリティの標準

[目的: バリデーション、認可、シークレット、データ保護のパターンを通じたセキュリティ姿勢の定義]

## 哲学
- 多層防御; 最小権限; デフォルトで安全（セキュアバイデフォルト）; フェイルクローズド
- 境界でバリデーションする; コンテキストに応じてサニタイズする; 入力を決して信頼しない
- 認証（誰か）と認可（何か）を分離する

## 入力と出力
- APIの境界とUIのフォームでバリデーションする; 型と制約を強制する
- 宛先（HTML, SQL, Shell, ログ）に応じてサニタイズ/エスケープする
- ブロックリストよりも許可リストを優先する; 最小限の詳細で早期にリジェクトする

## 認証と認可
- 認証: 身元を検証する; 短寿命のトークン/セッションを発行する
- 認可: アクションの前に権限をチェックする; デフォルトで拒否する
- ポリシーを一元化する; コード全体でのチェックの重複を避ける

パターン:
```typescript
if (!user.hasPermission('resource:action')) throw ForbiddenError();
```

## シークレットと設定
- シークレットを決してコミットしない; シークレットマネージャーまたは環境変数に保存する
- 定期的にローテーションする; アクセスを監査する; スコープを最小限にする
- 起動時に必須の環境変数をバリデーションする; 不足している場合は迅速に失敗させる

## 機密データ
- 収集を最小限にする; ログ内でマスク/秘匿化する; 保存時および転送時に暗号化する
- ロール/知る必要性（Need-to-Know）に基づいてアクセスを制限する; 機密性の高いレコードへのアクセスを追跡する

## セッション/トークンのセキュリティ
- 可能な場合は httpOnly + セキュア Cookie を使用する; 全面的な TLS
- 短い有効期限; リフレッシュ時にローテーションする; ログアウト時/漏洩時に失効させる
- トークンをオーディエンス/発行者にバインドする; 最小限のクレームを含める

## ログ記録（セキュリティを意識）
- 認証試行、権限拒否、機密性の高い操作をログに記録する
- パスワード、トークン、シークレット、完全な個人情報(PII)を決して記録しない; 完全なボディの記録を避ける
- イベントを関連付けるための requestId とコンテキストを含める

## ヘッダーと転送
- TLS を強制する; HSTS を使用する
- セキュリティヘッダーを設定する (CSP, X-Frame-Options, X-Content-Type-Options)
- 最新の暗号化方式を優先する; 脆弱なプロトコル/暗号スイートを無効にする

## 脆弱性への姿勢
- 安全なライブラリを優先する; 依存関係を常に更新する
- CIでの静的/動的スキャン; 追跡と修正を行う
- 一般的な攻撃クラスについてチームを教育する; それらを上記のパターンとしてエンコードする
