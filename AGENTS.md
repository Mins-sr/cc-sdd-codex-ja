# エージェント型 SDLC と仕様駆動開発

エージェント型 SDLC 上での Tp スタイルの仕様駆動開発

## プロジェクトメモリ
プロジェクトメモリは、Codex が各実行時に標準を遵守できるように、永続的なガイダンス（ステアリング、仕様メモ、コンポーネントドキュメント）を保持します。これをパターン、規則、決定事項のための長寿命な信頼できる情報源として扱ってください。

- プロジェクト全体の方針（アーキテクチャの原則、命名規則、セキュリティ制約、技術スタックの決定、API 標準など）には `.exec-plans/steering/` を使用してください。
- 機能やライブラリのコンテキスト（例: `src/lib/payments/AGENTS.md`）にはローカルの `AGENTS.md` ファイルを使用してください。該当フォルダに固有のドメインの前提事項、API コントラクト、またはテスト規則を記述します。Codex は一致するパスで作業する際にこれらを自動で読み込みます。
- 仕様レベルのワークフローをガイドするために、仕様メモは各仕様（`.exec-plans/specs/` 配下）と一緒に保持されます。

## プロジェクトコンテキスト

### パス
- ステアリング: `.exec-plans/steering/`
- 仕様: `.exec-plans/specs/`
- 製品コード: `apps/<product-name>/`

### ステアリング vs 仕様

**ステアリング** (`.exec-plans/steering/`) - プロジェクト全体のルールとコンテキストで AI をガイドする
**仕様** (`.exec-plans/specs/`) - 個別の機能の開発プロセスを形式化する

### ディレクトリルール
- ルート直下はエージェント運用、仕様、テンプレート、全体設定のために使用する
- 実際のブラウンフィールド製品コードは `apps/<product-name>/` 配下に配置する
- `myproduct` は説明用のプレースホルダーであり、実運用では実際の製品名を `kebab-case` で使用する
- C++ と Python が混在する場合も、まず製品単位で `apps/<product-name>/` にまとめ、その配下で言語別または既存構造を維持して整理する
- 製品固有のルールは `apps/<product-name>/AGENTS.md` またはその下位ディレクトリの `AGENTS.md` に記述する

### アクティブな仕様
- アクティブな仕様については `.exec-plans/specs/` を確認する
- 進捗を確認するには `$tp-spec-status [feature-name]` を使用する

## 開発ガイドライン
- 思考は英語、生成するレスポンスは日本語で行う。プロジェクトファイル（例: requirements.md, design.md, tasks.md, research.md, バリデーションレポート）に書き込まれるすべての Markdown コンテンツは、この仕様に設定されたターゲット言語（spec.json.language を参照）で記述されなければなりません。

## 最小ワークフロー
- フェーズ 0 (任意): `$tp-steering`, `$tp-steering-custom`
- ディスカバリー: `$tp-discovery "idea"` — アクションパスを決定し、複数仕様プロジェクトのために brief.md と roadmap.md を記述する
- フェーズ 1 (仕様):
  - 単一の仕様: `$tp-spec-quick {feature} [--auto]` またはステップバイステップ:
    - `$tp-spec-init "description"`
    - `$tp-spec-requirements {feature}`
    - `$tp-validate-gap {feature}` (任意: 既存のコードベース用)
    - `$tp-spec-design {feature} [-y]`
    - `$tp-validate-design {feature}` (任意: 設計レビュー)
    - `$tp-spec-tasks {feature} [-y]`
  - 複数仕様: `$tp-spec-batch` — 依存関係の波ごとに並行して roadmap.md からすべての仕様を作成する
- フェーズ 2 (実装): `$tp-impl {feature} [tasks]`
  - タスク番号なし: 自律モード (タスクごとのサブエージェント + 独立したレビュー + 最終バリデーション)
  - タスク番号あり: 手動モード (メインコンテキストで選択されたタスク、完了前に依然としてレビューアによるゲートあり)
  - `$tp-validate-impl {feature}` (独立した再バリデーション)
- 進捗チェック: `$tp-spec-status {feature}` (いつでも使用可能)

## スキル構造
スキルは `.agents/skills/tp-*/SKILL.md` に配置されています
- 各スキルは `SKILL.md` ファイルを持つディレクトリです
- 現在利用可能なスキルを検査するには `/skills` を使用する
- `$tp-<skill-name>` でスキルを直接呼び出す
- `tp-review` — レビューアのサブエージェントが使用するタスクローカルな対抗レビュープロトコル
- `tp-debug` — デバッガーのサブエージェントが使用する根本原因ファーストのデバッグプロトコル
- `tp-verify-completion` — 成功または完了の主張前の新たな証拠ゲート
- **現在のタスクにスキルが適用される可能性が1%でもある場合は、それを呼び出してください。** タスクが単純に見えるからといってスキルをスキップしないでください。

## コラボレーションモード (任意)
より長いタスクのために Codex が焦点の合った実行モードを選択できるように、`~/.codex/config.toml` でコラボレーションモードを有効にします:

```toml
[features]
collaboration_modes = true
```

## マルチエージェント (実験的)
マルチエージェントが利用可能な場合、それを使用してスキル内の独立した調査とバリデーションを並行化します。`~/.codex/config.toml` で有効にします:

```toml
[features]
multi_agent = true
```

「Parallel Research」セクションがあるスキルには、この機能がアクティブな場合にサブエージェントの生成が有益な独立した作業項目がリストされています。

## 開発ルール
- 3段階の承認ワークフロー: 要件 → 設計 → タスク → 実装
- 各フェーズで人間のレビューが必要。意図的なファストトラックの場合のみ `-y` を使用する
- ステアリングを最新の状態に保ち、`$tp-spec-status` との整合性を検証する
- ユーザーの指示に正確に従い、その範囲内で自律的に行動する: 必要なコンテキストを収集し、要求された作業をこの実行内でエンドツーエンドで完了させ、重要な情報が不足している場合や指示が致命的に曖昧な場合のみ質問する。

## ステアリング設定
- プロジェクトメモリとして `.exec-plans/steering/` 全体をロードする
- デフォルトファイル: `product.md`, `tech.md`, `structure.md`
- カスタムファイルがサポートされている (`$tp-steering-custom` 経由で管理)
