# Tp カスタムステアリング作成

<background_information>
**役割**: コアファイル（product, tech, structure）を超えた、専門的なステアリングドキュメントを作成すること。

**ミッション**: ユーザーが特定の専門領域においてドメイン固有のプロジェクトメモリを作成するのを支援すること。

**成功基準**:
- カスタムステアリングが専門的なパターンを捉えていること
- コアのステアリングと同じ粒度の原則に従っていること
- 特定のドメインに明確な価値を提供していること
</background_information>

<instructions>
### ステップ 1: コンテキストの収集

会話からすでにステアリングのコンテキストが利用可能な場合は、冗長なファイルの読み込みをスキップします。
それ以外の場合:
- 利用可能なテンプレートについて `.exec-plans/settings/templates/steering-custom/` を確認します
- ステアリングの原則として、このスキルディレクトリの `rules/steering-principles.md` を読み込みます

## ワークフロー

1. **カスタムステアリングのニーズをユーザーに尋ねる**:
   - ドメイン/トピック（例: "APIの標準", "テスト手法"）
   - 文書化すべき特定の要件やパターン

2. **テンプレートが存在するか確認する**:
   - 利用可能な場合は `.exec-plans/settings/templates/steering-custom/{name}.md` から読み込む
   - 出発点として使用し、プロジェクトに基づいてカスタマイズする

3. **関連するパターンのためにコードベースを分析する (JIT)**:

#### 並行調査

以下の調査領域は独立しており、並行して実行できます:
1. **テンプレートと原則**: 一致するテンプレートと steering-principles.md を読み込みます
2. **ドメインパターン**: Glob/Grep/Read を使用して、コードベース内のドメイン固有のパターンを分析します

マルチエージェントが有効な場合、上記の各領域に対してサブエージェントを生成します。それ以外の場合は順次実行します。

すべての並行調査が完了した後、ステアリングドキュメントのために調査結果を統合します。

4. **カスタムステアリングの生成**:
   - 利用可能な場合はテンプレートの構造に従う
   - このスキルディレクトリの `rules/steering-principles.md` の原則を適用する
   - 網羅的なリストではなく、パターンに焦点を当てる
   - 100〜200行（閲覧時間 2〜3 分）以内に収める

5. **ファイルを作成する**: `.exec-plans/steering/{name}.md`

## 利用可能なテンプレート

`.exec-plans/settings/templates/steering-custom/` にあるテンプレート:

1. **api-standards.md** - REST/GraphQL 規約、エラー処理
2. **testing.md** - テストの構成、モック、カバレッジ
3. **security.md** - 認証パターン、入力値のバリデーション、シークレット
4. **database.md** - スキーマ設計、マイグレーション、クエリパターン
5. **error-handling.md** - エラータイプ、ログ記録、再試行戦略
6. **authentication.md** - 認証フロー、権限、セッション管理
7. **deployment.md** - CI/CD、環境、ロールバック手順

必要な場合にテンプレートを読み込み、プロジェクトに合わせてカスタマイズします。

## ステアリングの原則

`rules/steering-principles.md`（このスキルのディレクトリ内）から:

- **リストよりもパターン**: すべてのファイル/コンポーネントではなく、パターンを文書化する
- **単一ドメイン**: ファイルごとに1つのトピック
- **具体的な例**: パターンをコードで示す
- **保守可能なサイズ**: 通常は100〜200行
- **セキュリティ・ファースト**: シークレットや機密データを含めない

</instructions>

## ツールガイダンス

- **Read**: テンプレートの読み込み、既存のコードの分析
- **Glob**: パターン分析のための関連ファイルを探す
- **Grep**: 特定のパターンを検索する
- **Bash** (`ls`を使用): 関連する構造を理解する

**JIT戦略**: そのタイプのステアリングを作成する場合にのみテンプレートを読み込む。

## 出力の説明

ファイルの場所を含むチャットでの要約 (ファイルは直接作成されます)。

```
✅ カスタムステアリングが作成されました

## 作成されたもの:
- .exec-plans/steering/api-standards.md

## 基準としたもの:
- テンプレート: api-standards.md
- 分析対象: src/api/ ディレクトリのパターン
- 抽出結果: REST の規約、エラーフォーマット

## 内容:
- エンドポイントの命名パターン
- リクエスト/レスポンスのフォーマット
- エラー処理の規約
- 認証のアプローチ

必要に応じて確認およびカスタマイズしてください。
```

## 例

### 成功例: APIの標準
**入力**: "APIの標準ステアリングを作成して"
**アクション**: テンプレートの読み込み、src/api/の分析、パターンの抽出
**出力**: プロジェクト固有のREST規約が含まれた api-standards.md

### 成功例: テスト戦略
**入力**: "テスト手法を文書化して"
**アクション**: テンプレートの読み込み、テストファイルの分析、パターンの抽出
**出力**: テストの構成とモック戦略が含まれた testing.md

## 安全性とフォールバック

- **テンプレートなし**: ドメイン知識に基づいてゼロから生成する
- **セキュリティ**: シークレットを決して含めない (原則を読むこと)
- **バリデーション**: コアのステアリングのコンテンツと重複していないことを確認する

## ノート

- テンプレートは出発点であり、プロジェクトに合わせてカスタマイズする
- コアのステアリングと同じ粒度の原則に従う
- すべてのステアリングファイルはプロジェクトメモリとして読み込まれる
- カスタムファイルはコアファイルと同じくらい重要である
- エージェント固有のツールディレクトリ (例: `.cursor/`, `.gemini/`, `.claude/`) の文書化は避ける
- `.exec-plans/specs/` および `.exec-plans/steering/` への軽い参照は許容される。他の `.exec-plans/` ディレクトリへの言及は避ける
