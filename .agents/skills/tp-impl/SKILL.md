# 実装スキル (tp-impl)

<background_information>
以下の2つのモードで動作します:
- **自律的(Autonomous)モード** (タスク番号なし): タスクごとに新しい(フレッシュな)サブエージェントをディスパッチし、各タスクの後に独立したレビューを行います
- **マニュアルモード** (タスク番号あり): 選択されたタスクをメインコンテキストで直接実行します

- **成功基準**:
  - 実装コードの前にすべてのテストが記述されていること
  - コードが退行（リグレッション）なくすべてのテストに合格すること
  - `tasks.md` でタスクに完了マークが付けられていること
  - 実装が設計および要件と一致していること
  - 完了の前に、独立したレビューアが各タスクを承認すること
</background_information>

<instructions>

## ステップ 1: コンテキストの収集

もし会話からステアリング/仕様のコンテキストが既に利用可能であれば、冗長なファイルの読み込みをスキップします。
そうでない場合は、必要なすべてのコンテキストを読み込みます:
- `.exec-plans/specs/{feature}/spec.json`, `requirements.md`, `design.md`, `tasks.md`
- コアステアリングコンテキスト: `product.md`, `tech.md`, `structure.md`
- 追加のステアリングファイルは、選択したタスクの境界、ランタイムの前提条件、統合、ドメインルール、セキュリティ/パフォーマンスの制約、または実装や検証に影響を与えるチーム規約に直接関係する場合にのみ読み込みます
- タスクのホスト環境またはユースケースに明確に一致する場合にのみ、関連するローカルエージェントスキルまたはプレイブック。ディレクトリ全体ではなく、必要な特定のアーティファクトのみを読んでください。

### 並行調査 (Parallel Research)

以下の調査領域は独立しており、並行して実行できます:
1. **仕様コンテキストの読み込み**: spec.json, requirements.md, design.md, tasks.md
2. **ステアリング、プレイブック、およびパターン**: コアステアリング、タスクに関連する追加ステアリング、一致するローカルエージェントのスキル/プレイブック、および既存のコードパターン

すべての並行調査が完了した後、開始する前に実装ブリーフ(implementation brief)を統合(synthesize)します。

### 事前確認 (Preflight)

**承認の検証**:
- spec.json でタスクが承認されていることを確認します（されていない場合は停止します。安全性とフォールバックを参照）

**検証コマンドの発見**:
- リポジトリローカルの「信頼できる情報源 (Source of Truth)」を次の順序で検査します: プロジェクトスクリプト/マニフェスト (`package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, アプリマニフェスト), タスクランナー (`Makefile`, `justfile`), CI/ワークフローファイル, 既存の e2e/統合環境設定, そして最後に `README*`
- このリポジトリの標準的な(canonical)検証セットを導出します: `TEST_COMMANDS`, `BUILD_COMMANDS`, および `SMOKE_COMMANDS`
- その場しのぎのシェルパイプラインよりも、リポジトリの自動化ですでに使用されているコマンドを優先します
- `SMOKE_COMMANDS` については、アプリの形態において最も軽くて信頼できる実行可能・生存確認（liveness check）を選択します（例: ルートURLの読み込み、Electronの起動、CLIの `--help` 指定、サービスのヘルスエンドポイント、すでに存在する場合はモバイルシミュレータ/e2e用ハーネス等）
- 全コマンドのセットは親コンテキストに保持し、タスクに関連するサブセットのみを実装者(implementer)およびレビューアサブエージェントに渡します。

**リポジトリベースラインの確立**:
- `git status --porcelain` を実行し、既存のコミットされていない変更があればメモします。

## ステップ 2: タスクの選択とモードの決定

**引数の解析**:
- `$1` から機能名を抽出する
- `$2` にタスク番号が指定されている場合 (例: "1.1" または "1,2,3"): **マニュアルモード**
- タスク番号がない場合: **自律的(Autonomous)モード** (保留中のすべてのタスク)

**タスクキューの構築**:
- tasks.md を読み、実行可能なサブタスク (1.1, 2.3 のような X.Y 番号付け) を特定する
- 主タスク (1., 2.) はグループ化のための見出しであり、実行単位ではありません
- `_Blocked:_` アノテーションが付いているタスクはスキップします
- 選択された各タスクについて、`_Depends:_` アノテーションを確認します -- 参照されているタスクが `[x]` になっているかを検証します
- 前提条件が未完了の場合は、先にそれらを実行するか警告を出します
- タスクのコンポーネント範囲(scope)を理解するために `_Boundary:_` アノテーションを使用します

## ステップ 3: 実装の実行

### 自律的(Autonomous)モード (サブエージェントのディスパッチ)

**反復（イテレーション）の規律**: 反復ごとに**正確に1つの**サブタスク (例: 1.1) のみを処理してください。単一のサブエージェントのディスパッチに複数のサブタスクをバッチ処理しないでください。各イテレーションは完全なサイクル（実装者のディスパッチ → レビュー → コミット → tasks.md の再読み込み → 次のタスク）に従います。

**コンテキストの管理**: 各反復の開始時に、`tasks.md` を再読して次の実行可能なサブタスクを決定します。以前の反復で蓄積されたメモリ（記憶）に依存してはなりません。各反復が完了した後、1行の要約 (例: "1.1: READY_FOR_REVIEW, 3 files changed") のみを保持し、完全なステータスレポートやレビューアーの詳細は破棄します。

マルチエージェント能力が利用可能な場合、各タスクについて (1つずつ):

**a) 実装者のディスパッチ**:
- このスキルのディレクトリから `templates/implementer-prompt.md` を読み込む
- テンプレートとタスク固有のコンテキストを組み合わせてプロンプトを構築します:
  - タスクの説明と境界範囲
  - 仕様ファイルへのパス: requirements.md, design.md, tasks.md
  - このタスクが満たさなければならない正確な要件と設計のセクション番号 (新しく捏造した `REQ-*` 型エイリアスではなく、ソースにある元のセクション番号を使用する)
  - タスクに関連するステアリングのコンテキストと、親が発見した検証コマンド (必要に応じて テスト/ビルド/スモーク(smoke))
  - タスクが振る舞いに影響を与えるものか (Feature Flag Protocolを使用)、または振る舞いに影響を与えないタスクであるか
  - **以前の学び (Previous learnings)**: このタスクの境界関係や依存関係に関連する tasks.md からの `## Implementation Notes` (実装メモ) のエントリを含めます (例: "better-sqlite3 requires separate rebuild for Electron")。これにより同じ間違いの再発を防ぎます。
- 実装を始める前に、実装者(implementer)・サブエージェントは仕様ファイルを読み込んで独自の「タスクブリーフ（Task Brief）」(受け入れ基準、完了定義、設計の制約、検証方法) を構築します。
- このプロンプトで新しい(fresh)サブエージェントを生成します

**b) 実装者ステータスの処理**:
- 実装者のステータスは、正確な `## Status Report` ブロックと `- STATUS:` フィールドからのみ解析します。
- `STATUS` がない、曖昧である、または散文（長い文章）に置き換えられている場合は、正確な構造化されたステータスブロックのみを要求して、実装者を1回だけ再ディスパッチします。解析可能な(`READY_FOR_REVIEW | BLOCKED | NEEDS_CONTEXT`) なしにはレビューに進んではいけません。
- **READY_FOR_REVIEW** → レビューに進む
- **BLOCKED** → デバッガーサブエージェントをディスパッチします (下のセクションを参照); すぐにスキップしないでください
- **NEEDS_CONTEXT** → 要求された追加のコンテキストを使用して1回再ディスパッチします。 それでも解決しない場合 → デバッガーサブエージェントをディスパッチします

**c) レビューアーのディスパッチ**:
- このスキルのディレクトリから `templates/reviewer-prompt.md` を読み込みます
- 以下を使用してレビュープロンプトを構築します:
  - タスクの説明と関連する仕様のセクション番号
  - レビューアーがそれらを直接読めるようにするための仕様ファイル (requirements.md, design.md) へのパス
  - 実装者のステータスレポート (参照用のみです — レビューアーは独自に検証する必要があります)
- レビューアーはこのタスクローカルなレビューに `tp-review` プロトコルを適用しなければなりません。
- 既存のタスク固有のコンテキストを維持します: タスクのテキスト、仕様の参照、`_Boundary:_` 範囲、検証コマンド、実装者のレポート、および一次情報源としての実際の `git diff`。
- レビューアーのサブエージェントは自身で `git diff` を実行して、実際のコード変更を読み取り、仕様に対する検証を行います
- このプロンプトで新しい(fresh)サブエージェントを生成します

**d) レビューアーの判定(verdict)の処理**:
- レビューアーの判定(verdict)は、正確な `## Review Verdict` ブロックと `- VERDICT:` フィールドからのみ解析します。
- もし `VERDICT` が無いか、曖昧か、散文に置き換わっている場合は、正確な構造化された判定のみを要求してレビューアーを１回再ディスパッチします。解析可能な `APPROVED | REJECTED` 値がないのにタスクを完了としたり、コミットしたり、次のタスクに進んだりしてはいけません。
- **APPROVED** → タスクに `[x]` マークを付けたり成功を主張したりする前に、現在のコードの状態からの新しい証拠を使用して `tp-verify-completion` を適用します。その後、tasks.md にてこのタスクに `[x]` マークをつけ、部分（選択的）な Git コミットを実行します。
- **REJECTED (第1-2ラウンド)** → レビューフィードバックを含めて実装者を再ディスパッチします
- **REJECTED (第3ラウンド)** → デバッガーサブエージェントをディスパッチします (下のセクションを参照)

**e) コミット** (親エージェントのみ, 選択的ステージング):
- このタスクのために実際に変更されたファイルと tasks.md のみをステージングします
- `git add -A` や `git add .` は **絶対に使用しない** でください
- 明示的なファイルパスを使用して `git add <file1> <file2> ...` を使用します
- コミットメッセージの形式: `feat(<feature-name>): <task description>`

**f) 学び（ルールや知見）の記録**:
- このタスクによって横断的な洞察が明らかになった場合は、tasks.md の下部にある `## Implementation Notes` セクションに一行のコメントを追記します。

**g) デバッグ サブエージェント** (BLOCKED、NEEDS_CONTEXT の未解決、または２回の修正ラウンドを経ても REJECTED の場合にトリガーされる):

デバッグサブエージェントは **フレッシュなコンテキスト** で実行されます — エラー情報のみを受け取り、失敗した実装の全履歴は受け取りません。これにより、無限の再試行ループを引き起こすコンテキストの汚染（pollution）を回避できます。

- このスキルのディレクトリから `templates/debugger-prompt.md` を読み込みます
- 次の情報でデバッグプロンプトを構築します:
  - エラーの説明 / ブロックされた理由 / レビューアーの拒否理由 (findings)
  - コミットされていない現在の変更をまとめた `git diff`
  - タスクの説明と関連する仕様のセクション番号
  - デバッガーが仕様ファイルを参照できるよう各仕様ファイルへのパス
- デバッガーはこの失敗の調査（investigation）に `tp-debug` プロトコルを適用しなければなりません。
- リッチ(豊富)な失敗コンテキストを保持します：エラー出力、レビューアーによる発見事項(findings)、現在の `git diff`、タスクおよび仕様の参照先、関連する実装ノート (Implementation Notes)。
- 可能な場合、デバッガーは実行時状態（ランタイム状態）/設定状態を検査し、修正計画(fix plan)を提案する前に、Web または公式ドキュメントからのリサーチを用いて根本原因の仮説を検証(validate)してください。
- このプロンプトで新しい(fresh)サブエージェントを生成します

**デバッグ レポートの処理**:
- デバッグレポートの正確な構造化されたフィールドから `NEXT_ACTION` を解析します。
- もし `NEXT_ACTION: STOP_FOR_HUMAN` の場合 → tasks.md に `_Blocked: <ROOT_CAUSE>_` と追記し、この機能(feature)の実行を停止（中断）し、続行する前に人によるレビューが必要であることを報告してください。
- もし `NEXT_ACTION: BLOCK_TASK` の場合 → tasks.md に `_Blocked: <ROOT_CAUSE>_` を追記し、次のタスクへスキップします
- もし `NEXT_ACTION: RETRY_TASK` の場合 → 現在のワークツリー(作業中の状態)を維持します。無関係な変更をリセットまたは破棄**しないでください**。デバッグレポートから抽出した `FIX_PLAN`、`NOTES`、および現在の `git diff` とともに**新しい(new)**実装者（implementer）サブエージェントを生成し、明示的な変更のみで該当タスクを修復することを要求します
  - 新しい実装者が成功した場合（READY_FOR_REVIEW → reviewers にて APPROVED） → 通常のフロー
  - 新しい実装者も失敗した場合 → デバッグサイクルを繰り返す (合計で最大2回のデバッグラウンド)。2回失敗した場合 → tasks.mdに `_Blocked: debug attempted twice, still failing — <ROOT_CAUSE>_` を追記してスキップします。
- **1つのタスクにつき最大2デバッグラウンド**: ラウンドごと： 新しいデバッグ・サブエージェント → 新しい実装者。それでも失敗する場合はそのタスクをブロック済(blocked)とします。
- デバッグで見つかった知見を `## Implementation Notes` に記録します（これにより、後続のタスクで同じ問題を踏まないようにするのに役立ちます)

**`(P)` マーカー**: tasks.md において `(P)` のマークが付いたタスクは、相互依存関係がなく、理論的には並行して実行できることを示します。ただし、tp-impl では Git の競合（コンフリクト）を避け、レビューを簡素化するためにそれらを逐次実行(一つずつ実行)します。 `(P)` マーカーはタスク計画のための情報目的のものであり、実行の指示(ディレクティブ)ではありません。

**フォールバック**: マルチエージェントが利用できない場合、すべてのタスクについてマニュアルモードが実行されます。

### マニュアルモード (メインコンテキスト)

選択した各タスクについて:

**1. タスク概要 (Task Brief) の構築**:
コードを書き始める前に、requirements.md と design.md の該当タスクの関連セクションを読み、以下の点を明確にします:
- （完了時に）どのような観察可能な振る舞いがなされているべきか (受入基準)
- どのファイル/関数/テストが存在しなければならないか (完了の定義)
- design.md からどのような技術的決定を踏襲するべきか (設計上の制約)
- そのタスクが動作することをどのように確認するか (検証方法)

**2. TDDサイクルの実行** (Kent Beck による RED → GREEN → REFACTORのサイクル):
- **RED**: 受入基準に基づいて、次の小さな機能のためのテストを記述します。テストは**失敗(Fail)**すること。
- **GREEN**: 設計の制約に従い、テストに合格(Pass)するための最もシンプルな解決策を実装します。
- **REFACTOR (リファクタリング)**: コードの構造を改善し、重複を排除します。すべてのテストに合格し続ける(Pass)必要があります。
- **VERIFY (検証)**: すべてのテスト(新規および既存のテスト)がパスし、デグレ(regression)が生じていないことを確認します。検証方法によって確認できていることを確実にして下さい。
- **REVIEW**: タスクを完了とする前に `tp-review` を適用します。ホストがマニュアルモードでフレッシュな(fresh)サブエージェントを利用可能な場合はフレッシュなレビューアーを使用し、そうでない場合はメインコンテキストで `tp-review` プロトコルを使用してレビューを実行します。判定が明確に `APPROVED` となるまでは次に進まないこと。
- **MARK COMPLETE (完了のマーク)**: レビューアーが `APPROVED` を返した後に**のみ**、`tp-verify-completion` を適用し、その後で tasks.md のチェックボックスを `- [ ]` から `- [x]` に変更します。

## ステップ 4: 最終的な「承認(Validation)」

**自律的(Autonomous)モード**:
- すべてのタスクが完了した後、GO/NO-GO を判定するゲートとして `$tp-validate-impl $1` を実行します。
- バリデーションが GO を返す場合 → この機能の完了を宣言(success claim)する前に、バリデーションの結果と新しい証拠データを使って機能レベルの主張に対して `tp-verify-completion` を適用します。
- もしバリデーションが NO-GO を返した場合:
  - バリデーションレポートからの具体的な問題（findings）のみを修正してください。
  - 修正対応(remediation)は 3 回までに制限します。それでも NO-GO のままの場合は、実行を停止し、残りの見つかった問題を報告します。
- バリデーションから MANUAL_VERIFY_REQUIRED が返ってきた場合 → 停止し不足している検証手順を報告します。

**マニュアルモード**:
- `$tp-validate-impl $1` を実行することを提案しますが、自動実行はしないでください。

## フィーチャー・フラグ（Feature Flag）プロトコル

振る舞いを追加または変更するタスクについては、フィーチャー・フラグを使用してRED → GREEN を強制します:

1. **フラグの追加** (デフォルトは OFF): コードベースに適した切り替え（トグル）機能（環境変数、設定の定数、ブール値、条件分岐など）を導入します。
2. **RED -- フラグ OFF**: 新しい振る舞いに対するテストを記述します。テストを実行 → 失敗 (FAIL) すること。フラグが OFF の状態でテストが通ってしまう場合、そのテストは正しいものを検証していません。書き直してください。
3. **GREEN -- フラグ ON + 実装**: フラグを有効(ON)にし、実装を記述します。テストを実行 → 合格 (PASS) すること。
4. **フラグの削除**: コードから(フラグに依る)条件分岐を外し、恒常的なものとします。テストを実行 → 引き続き合格 (PASS) すること。

**注意:** リファクタリング、設定、ドキュメントの作成、または振る舞いに変化がないタスクの場合はこのプロトコルをスキップしてください。

</instructions>

## 重要な制約事項
- **厳密なハンドオフ解析**: 実装者の `STATUS` や サブエージェントによる `VERDICT` を周辺の長い文章(散文)から推測することは絶対に避けてください。正確な構造での（決められた）フィールドのみが評価対象となります。
- **破壊的リセットの禁止**: 実装のループの内部で、`git checkout .`、`git reset --hard`、またはそれらと同等の破壊的なロールバックを決して使用しないでください。
- **選択的ステージング**: `git add -A` または `git add .` は決して試用しないでください(NEVER)。常に明示的なファイルパスからステージング(git add)を行ってください。
- **限定的なレビュー回数制限**: 実装者の再ディスパッチの回数は「1回のレビューアによるRejection(リジェクト)」対し最大で2回までです。その後はデバッグへ移ります。
- **上限のあるデバッグ**: 各タスクで最大2回のデバッグ（1ラウンドの中に「デバッグ」＋「再実装」）が行われます。それでも機能しない場合は、該当箇所のステータスは BLOCKED (実装保留中)になります。
- **上限のある修正作業**: (一通りの機能を作った後の)「最終バリデーション時の再修正」は3回までに制限します。

## 出力の説明

**自律的(Autonomous)モード**:
各タスクについて以下を報告します:
タスク ID、実装者のステータス、レビューアの判定、変更されたファイル、コミットハッシュ。
すべてのタスクが終わった後: 最終の検証結果。

**マニュアルモード**:
実行されたタスクとそののテスト結果。完了／未完了タスクのステータス。

**形式(フォーマット)**: 簡潔に。使用言語は spec.json で指定されている（ターゲット）言語とします。

## 安全性とフォールバック

### エラーシナリオ

**タスクが承認されていない、または仕様文書が見つからない場合**:
- **実行停止**: 全要件が揃い、すべての仕様文書が承認済である必要があります。
- **推奨されるアクション**: まず、以前のフェーズとして完了させること。実行例: `$tp-spec-requirements`, `$tp-spec-design`, `$tp-spec-tasks`。

**テストでの失敗（フェイル）**:
- **実装の停止**: 不合格となっているテストの修正を優先します。（開発の続行より先にテストのデバッグに取り組む）
- **アクション**: デバッグと修正を終えたあとで再度実行します。

**すべてのタスクがブロックされている(BLOCKED)場合**:
- 各問題の理由を示し、実行の強制的な停止と人間(ヒト)による精査を促します。

**仕様と現実での矛盾(コンフリクト)が生じているケース**:
- サイレントになんとか解決しようと避けず、このタスクを「`_Blocked: <reason>_`」扱いで明確に保留します。

**上流への帰属が判明した場合** (Upstream Ownership Detected):
- このタスクのレビュー時、デバッグ時あるいは検証で問題要因に遡る根本的な原因(root causes)が、「上流・基礎構造・共有基盤」側もしくは「依存する旧プロジェクトの使用」にあると判明したときは、現行プロジェクト下で場当たり的な一時修正での「応急対応実装(パッチ)」をおこなってはいけません。
- かわりに直接要因を持つ関連先側の「仕様(Specs)」に対して修正を移し、現側(Downstream = 下流タスク)はこの関連元の確約(Contract)が完全に回復・担保される間、保留された「Blocked」対応にする必要があります。また、元側の改修着手後も再度「Downstream向けの確認・テスト稼働(smoke test/validation)」をやり直します。

**実装中にタスク計画が無効になった場合**:
- もしタスクの順番（順序付け）、境界、または分割(decomposition)での問題によってデバッグが `NEXT_ACTION: STOP_FOR_HUMAN` を返した場合は、コードでの解決（ワークアラウンド）を強制するのではなく、直ちに中断して `tasks.md` における（あるいは承認済みの計画上の）構成の変更（Human による判断）に戻します。

**セッションが中断された場合**:
- 中断されたタスクについては、タスクが tasks.md 配下では完了扱いの「`[x]`」となり、「gitにコミット済みの実装分」として残るため、再度 `$tp-impl $1` を再開することが安全です。
- 実行役であるコントローラーは（繰り返しの各）イテレーションの単位毎で `tasks.md` 内の状態を最新に把握(Re-Reads)しているため、前回の完了時、すなわち（自動的に）直前どこまで完了が行われて終了していた時点からの、安全な続きでの自動処理復帰がされます。
