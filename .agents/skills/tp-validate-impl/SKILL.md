# 実装の統合の検証 (tp-validate-impl)

<background_information>
個々のタスクは、実装中にタスクごとのレビューアによってすでにレビューされています。あなたの仕事は、すべてのタスクを一緒に見たときにしか現れない問題を捕まえることです。

境界（Boundary）に関する用語の連続性：
- ディスカバリーは `Boundary Candidates` (境界の候補) を特定します
- デザイン(設計)は `Boundary Commitments` (境界の確約) を固定します
- タスクは `_Boundary:` にて実行を制約します
- 機能の検証(Feature validation)はタスクを越えた `Boundary Violations` (境界の違反) をチェックします

- **成功基準**:
  - tasks.md 内のすべてのタスクに `[x]` がマークされている
  - 完全なテストスイートに合格している（タスクごとのテストだけでなく）
  - タスク間の統合が機能している（コンポーネント間でデータが流れ、インターフェースが一致している）
  - 要件のカバー率がすべてのタスク全体で完全である（タスク間にギャップがない）
  - 設計の構造がエンドツーエンドで反映されている（コンポーネント単位だけでなく）
  - 孤立したコード、競合する実装、統合時の継ぎ目(seams)での問題、または境界からの流出(spillover)がない

**このスキルが「やらないこと」**: タスクごとのチェックは `$tp-impl` 実行中のレビューアの責任です。このスキルは、個々のタスクの受け入れ基準、ファイルごとの確認（リアリティチェック）、または単一タスクの仕様との整合性を再チェックするものではありません。

このスキルの主な質問は「完了したタスクをまとめて見たとき、それらは依然として設計された境界の継ぎ目と依存関係の方向を尊重しているか？」です。
</background_information>

<instructions>
## 実行手順

### 1. 検証対象の検出

**引数が指定されていない場合** (`$1` が空):
- `$tp-impl <feature> [tasks]` コマンドの会話履歴を解析する
- 各実行から機能名とタスク番号を抽出する
- 実装されたすべてのタスクを機能ごとに集約する
- 検出された実装を報告する（例: "user-auth: 1.1, 1.2, 1.3"）
- 履歴が見つからない場合は、完了したタスク `[x]` のある機能を探すために `.exec-plans/specs/` をスキャンする

**機能が指定されている場合** (`$1` あり、`$2` が空):
- 指定された機能を使用する
- `.exec-plans/specs/$1/tasks.md` 内のすべての完了タスク `[x]` を検出する

**機能とタスクの両方が指定されている場合** (`$1` と `$2` があり):
- 指定された機能とタスクのみを検証する（例: `user-auth 1.1,1.2`）

#### サブエージェントのディスパッチ (並行処理)

以下の検証の切り口（ディメンション）は独立しており、**サブエージェント**としてディスパッチできます。機能の範囲に基づいて、サブエージェントを分割、結合、またはスキップするかどうか、最適な分解を決定してください。メインのコンテキストをGO/NO-GOの統合評価用にクリーンに保つため、各サブエージェントは**構造化された問題の要約(findings summary)**を返します。

**典型的な検証の切り口** (適宜調整すること):
- **テストの実行**: 完全なテストスイートを実行し、合否を詳細とともに報告する
- **要件のカバー率**: 要件 → 実装のマトリックスを構築し、ギャップを報告する
- **設計の整合性**: アーキテクチャが design.md と一致していることを検証し、ドリフト（逸脱）や依存関係の違反を報告する
- **タスクを越えた統合**: データフロー、API契約、共有状態の一貫性を検証する

マルチエージェントが利用できない場合は、メインコンテキストで順次チェックを実行します。

すべてのチェックが完了したら、GO/NO-GO/MANUAL_VERIFY_REQUIRED の評価のために問題(findings)を統合します。

### 2. コンテキストの読み込み

検出された各機能について:
- メタデータのために `.exec-plans/specs/<feature>/spec.json` を読み込む
- 要件のために `.exec-plans/specs/<feature>/requirements.md` を読み込む
- 設計の構造のために `.exec-plans/specs/<feature>/design.md` を読み込む
- タスクリストと実装ノート（Implementation Notes）のために `.exec-plans/specs/<feature>/tasks.md` を読み込む
- コアのステアリングコンテキスト: `product.md`, `tech.md`, `structure.md`
- 追加のステアリングファイルは、検証される境界、ランタイムの前提条件、統合、ドメインルール、セキュリティ/パフォーマンスの制約、または GO/NO-GO の判断に影響を与えるチームの規約に直接関係する場合にのみ読み込む

**標準的な(canonical)バリデーションコマンドの発見**:
- リポジトリローカルの「信頼できる情報源 (Source of Truth)」を次の順序で検査する: プロジェクトスクリプト/マニフェスト (`package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, アプリマニフェスト), タスクランナー (`Makefile`, `justfile`), CI/ワークフローファイル, 既存の e2e/統合環境設定, そして最後に `README*`
- このリポジトリにおける機能レベルの検証セットを見つけ出す: `TEST_COMMANDS`, `BUILD_COMMANDS`, および `SMOKE_COMMANDS`
- その場しのぎのシェルパイプラインよりも、リポジトリの自動化ですでに使用されているコマンドを優先する
- `SMOKE_COMMANDS` については、アプリの形態において最も軽くて信頼できる実行可能・生存確認（liveness check）を選択する（例: ルートURLの読み込み、Electronの起動、CLIの `--help` 定示、サービスのヘルスエンドポイント、すでに存在する場合はモバイルシミュレータ/e2e用ハーネス等）
- 複数の候補が存在する場合は、セットアップコストが最も小さく、それでも実際にビルドされた成果物を動かせるコマンドを優先する

### 3. 統合バリデーションの実行

#### 機械的チェック (コマンドを実行し、結果を使用する)

**A. 完全なテストスイート**
- 発見された標準的な完全なテストコマンドを実行する。終了コードを使用する。
- テストが失敗した場合 → NO-GO。ここで判断は不要。
- 標準的なテストコマンドが特定できない場合 → `MANUAL_VERIFY_REQUIRED`

**B. 残留 TBD/TODO/FIXME**
- 次を実行: `grep -rn "TBD\|TODO\|FIXME\|HACK\|XXX" <files-in-feature-boundary>`
- この機能によって導入された一致が見つかった場合 → 警告 (Warning) としてフラグを立てる

**C. 残留しているハードコードされたシークレット**
- 次を実行: `grep -rn "password\s*=\|api_key\s*=\|secret\s*=\|token\s*=" <files-in-feature-boundary>` (大文字小文字を区別しない)
- 環境変数の参照ではない一致が見つかった場合 → 致命的 (Critical) としてフラグを立てる

**D. ランタイムの生存確認 (Smoke Boot)**
- 構築された成果物が実際に起動し、最初の使用可能な状態に達することを証明する、発見された標準的なスモークコマンドを実行する。
- 関連する場合の例: ヘッドレスブラウザでルートURLを開き、起動時のコンソールエラーがゼロであることを要求する。Electronを起動し、メインプロセスの準備完了シグナルと最初のレンダラー読み込みを待つ。 `--help` オプションを付けてCLIを実行する。サービスを開始し、そのヘルスエンドポイント(死活監視)にアクセスする。
- 起動によりランタイムクラッシュ、処理されない例外、モジュール読み込みの失敗、ネイティブABIの不一致、または必要な環境/設定の欠落が発生した場合 → NO-GO。
- 信頼できるスモークコマンドを特定できない場合、または必要なランタイム環境が利用できない場合 → `MANUAL_VERIFY_REQUIRED`

#### 判断によるチェック (コードを読み、仕様と比較する)

**E. クロスタスク（複数のタスクにまたがる）統合**
- タスク間でインターフェイス、データモデル、API契約（コントラクト）を共有している箇所を特定する
- タスクAの出力フォーマットとタスクBの想定入力が一致しているか検証する
- タスク間で矛盾する想定（命名規則、エラーコード、データ構造）がないか確認する
- 共有状態（データベーススキーマ、設定、環境設定）がタスク間で一貫しているか検証する
- ある境界の振る舞いを別の境界に漏らす(leak)ことによってではなく、意図された継ぎ目（seams）で統合が行われているかを検証する

**F. 要件網羅率（カバレッジ）のギャップ**
- すべての要件セクションが少なくとも1つの完了したタスクにマッピングされていること
- 1つのタスクだけでは完全にカバーされていない要件（横断的要件）を特定する
- 複数のタスクによって部分的にカバーされているが、どのタスクにも完全にカバーされていない要件を特定する
- `requirements.md` の元のセクション番号を使用すること。 `REQ-*` のようなエイリアスを勝手に発明（捏造）しないこと

**G. 設計のエンドツーエンドの整合性**
- 全体的なコンポーネントグラフが design.md と一致していることを検証する
- 統合パターン（イベントフロー、API 境界、依存性の注入）が設計通りに機能することを確認する
- 依存関係の方向が design.md のアーキテクチャに従っていること（上方向へのインポートがないこと）を検証する
- ファイル構造計画（File Structure Plan）が実際のファイルレイアウトと一致していることを検証する
- 元の設計からのアーキテクチャ上のドリフト（逸脱）を特定する
- `design.md` の元のセクション番号を使用すること

**G.5 境界の監査**
- 完了した作業と、設計の「境界の確約」(Boundary Commitments)、「境界外」(Out of Boundary)、「許可された依存関係」(Allowed Dependencies)、および「再検証トリガー」(Revalidation Triggers)を比較する
- あるエリアが別の境界の責任をひそかに吸収しているような、タスクを越えた流出(spillover)を特定する
- 「統合を簡単にするため」という理由で上流(upstream)に埋め込まれた、下流(downstream)固有の回避策(ワークアラウンド)を特定する
- 設計で宣言されていなかった、新しい隠れた依存関係または共有所有権を特定する
- もし再検証のトリガーが引かれた場合、影響を受ける隣接する仕様または統合ポイントが実際に再チェックされたことを検証する

**H. ブロックされたタスクと実装ノート**
- `_Blocked:_` とマークされているタスクが今もないか確認する — なぜかを報告し、機能の完全性への影響を評価する
- tasks.md 内の `## Implementation Notes` を確認し、注意を要する横断的な洞察を探す

### 4. レポートの生成

`GO` を返す前に、`tp-verify-completion` プロトコルを機能レベルの主張（claim）に適用してください。テストだけでは不十分です： エビデンスに完全なスイート、ランタイムのliveness、カバレッジ、統合、設計の整合性、ブロックされたタスクのステータスを含めてください。

修正案（remediation）を記述する前に、所有権（ownership）によって具体的な失敗を分類します：
- `LOCAL` (ローカル) : 欠陥が検証対象の機能に属する場合
- `UPSTREAM` (アップストリーム) : 根本原因が依存関係、基盤、共有プラットフォーム、または以前の仕様に属する場合
- `UNCLEAR` (不明瞭) : 利用可能な情報からは所有権を確立できない場合

所有権が `UPSTREAM` の場合、その問題をこの機能のローカルな修正に押し込めないでください。所有する上流の仕様を指定し、その上流の修正が反映された後、どの依存仕様が再検証されるべきかを説明してください。

spec.json で指定された言語でサマリーを提供してください:

```
## バリデーション レポート
- DECISION: GO | NO-GO | MANUAL_VERIFY_REQUIRED
- MECHANICAL_RESULTS:
  - Tests: PASS | FAIL (コマンドと終了コード)
  - TBD/TODO grep: CLEAN | <該当数> matches
  - Secrets grep: CLEAN | <該当数> matches
  - Smoke boot: PASS | FAIL | MANUAL_REQUIRED
- INTEGRATION:
  - Cross-task contracts (タスク間の契約): <ステータス>
  - Shared state consistency (共有される状態の一貫性): <ステータス>
  - Boundary audit (境界の監査): <ステータス>
- COVERAGE:
  - Requirements mapped (マッピングされた要件): <X/Y のセクションをカバー>
  - Coverage gaps (カバー率のギャップ): <カバーされていない要件セクションのリスト>
- DESIGN:
  - Architecture drift (アーキテクチャの逸脱): <発見や指摘事項>
  - Dependency direction (依存関係の方向): <もしあれば違反事項>
  - File Structure Plan vs actual (ファイル構造計画との差異): <一致/不一致>
- OWNERSHIP: LOCAL | UPSTREAM | UNCLEAR
- UPSTREAM_SPEC: <機能名 | N/A>
- BLOCKED_TASKS: <リストと影響の評価>
- REMEDIATION: <NO-GOの場合: 各問題を修正するための具体的かつ実行可能なステップ>
```

NO-GO の場合、REMEDIATION（修正事項）は必須です。正確な問題と何を変更する必要があるかを特定してください。

## 重要な制約
- **厳格な最終ゲート**: すべての統合チェックに合格した場合にのみ `GO` を返し、具体的な失敗がある場合は `NO-GO` を返し、必須の検証が完了できなかった場合は `MANUAL_VERIFY_REQUIRED` を返します
- **利便性よりも境界の完全性を優先**: たとえテストに合格したとしても、境界の間に責任を塗りつけることによってしか機能しない場合は、`GO` を返さないでください
</instructions>

## 安全性とフォールバック

### エラーシナリオ
- **実装が見つからない**: 履歴に `$tp-impl` がなく、タスクで `[x]` になっているものもない場合、「No implementations detected (実装が検出されませんでした)」と報告します
- **テストコマンドが不明**: `MANUAL_VERIFY_REQUIRED` を返し、どのバリデーションコマンドが欠落しているか説明します。`GO` は返さないでください
- **仕様ファイルの欠落**: spec.json/requirements.md/design.md が欠落している場合、エラーで停止します

### 次のステップのガイダンス

**GO判定の場合**:
- 機能はエンドツーエンドで検証され、デプロイメントまたは次の機能への準備が整いました

**NO-GO判定の場合**:
- リストされた統合の問題に対処します
- ターゲットを絞った修正のために `$tp-impl <feature> [tasks]` を再実行します
- `$tp-validate-impl [feature]` で再検証します

**セッションが中断された場合**:
- 再実行しても安全です — 検証は読み取り専用で冪等（idempotent）です

**MANUAL_VERIFY_REQUIRED の場合**:
- 機能を完了したものとして扱わないでください
- 正確な足りない検証ステップ、または環境の前提条件を提示してください
