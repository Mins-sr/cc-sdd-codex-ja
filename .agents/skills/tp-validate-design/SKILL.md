# 技術設計のバリデーション

<background_information>
- **ミッション**: 実装の準備が整っていることを確認するため、技術設計のインタラクティブな品質レビューを実施する
- **成功基準**:
  - 重大な問題が特定されること（最大3つの最も重要な懸念事項）
  - 強みも認識された、バランスの取れた評価であること
  - 理由を伴った明確な GO/NO-GO の決定があること
  - 必要に応じて、改善のための実行可能なフィードバックを提供すること
</background_information>

<instructions>
## コアタスク
承認された要件と設計ドキュメントに基づき、機能 **$1** の設計品質についてインタラクティブなレビューを実行します。

## 実行手順

1. **コンテキストの収集**:
   - 言語とメタデータについて `.exec-plans/specs/$1/spec.json` を読み込む
   - 要件について `.exec-plans/specs/$1/requirements.md` を読み込む
   - 設計内容について `.exec-plans/specs/$1/design.md` を読み込む
   - コアのステアリングコンテキスト: `product.md`, `tech.md`, `structure.md`
   - 追加のステアリングファイルは、アーキテクチャの境界、統合、ランタイムの前提条件、ドメインルール、セキュリティ/パフォーマンスの制約、または実装準備に影響を与えるチーム規約に直接関係する場合にのみ読み込む
   - 関連するローカルエージェントのスキルやプレイブックは、機能のホスト環境やユースケースに明確に一致し、レビューに関連するコンテキストを提供する場合にのみ読み込む

#### 並列調査

以下の調査領域は独立しており、並行して実行できます:
1. **コンテキストとルールの読み込み**: レビュー基準のために、仕様ドキュメント、コアステアリング、タスクに関連する追加ステアリング、関連するローカルエージェントのスキル/プレイブック、およびこのスキルディレクトリの `rules/design-review.md` を読み込む
2. **コードベースのパターン調査**: 既存のアーキテクチャパターン、命名規則、コンポーネント構造をコードベースから収集し、レビュー中の参照として使用する

マルチエージェントが有効な場合、上記の各領域に対してサブエージェントを生成します。それ以外の場合は順次実行します。

すべての並行調査が完了した後、調査結果を統合してレビューを行います。

2. **設計レビューの実行**:
   - 可能な場合は会話履歴を参照する: 以前の要件に関する議論やユーザーが述べていた設計意図を活用する
   - design-review.md のプロセスに従う: 分析 → 重大な問題 → 強み → GO/NO-GO
   - 最も重要な懸念事項は最大3つまでに制限する
   - ユーザーとインタラクティブに関わる — 確認の質問をし、代替案を提案する
   - 出力には spec.json で指定された言語を使用する

3. **決定と次のステップ**:
   - 理由を伴う明確な GO/NO-GO 決定
   - 具体的で実行可能な次のステップを提供する（以下の「次のフェーズ」を参照）

## 重要な制約
- **完全性の追求ではなく、品質の保証**: 許容可能なリスクは受け入れる
- **重大な焦点のみ**: 最大3つの問題に絞り、成功に大きく影響するもののみとする
- **会話への意識**: 可能な場合、要件のコンテキストやユーザーの意図を理解するために議論履歴を活用する
- **インタラクティブなアプローチ**: 対話に参加し、確認の質問をし、代替案を提案する
- **バランスの取れた評価**: 強みと弱みの両方を認識する
- **実行可能なフィードバック**: 提案はすべて実装可能でなければならない
- **コンテキストの規律**: コアステアリングから開始し、レビューに関連するステアリングやユースケースに沿ったローカルエージェントのスキル/プレイブックのみを拡張する
</instructions>

## ツールガイダンス
- **最初に読む**: レビューの前に、仕様、コアステアリング、関連するローカルプレイブック/エージェントスキル、およびルールをロードする
- **必要に応じて Grep**: パターンの検証や統合の確認のためにコードベースを検索する
- **インタラクティブ**: レビュープロセス全体を通じてユーザーと関わる

## 出力の説明
spec.json で指定された言語を使用して、以下の内容を含む出力を提供してください:

1. **レビューの要約**: 設計の品質と準備状況の簡潔な概要（2〜3文）
2. **重大な問題**: 最大3つ。design-review.md の形式に従うこと
3. **設計の強み**: 1〜2つのポジティブな側面
4. **最終評価**: 理由と次のステップを伴う GO/NO-GO 決定

**フォーマット要件**:
- 明確さのために Markdown の見出しを使用する
- design-review.md の出力形式に従う
- 要約は簡潔に保つ

## 安全性とフォールバック

### エラーシナリオ
- **設計の欠落**: design.md が存在しない場合は、メッセージ「設計ドキュメントを生成するために先に `$tp-spec-design $1` を実行してください」を表示して停止する
- **設計が生成されていない**: spec.json で設計フェーズが「生成済み」としてマークされていない場合は警告するが、レビューは続行する
- **空のステアリングディレクトリ**: プロジェクトコンテキストが不足しており、レビュー品質に影響を与える可能性があることをユーザーに警告する
- **言語未定義**: spec.json で言語が指定されていない場合は、デフォルトで英語 (`en`) となる

### 次のフェーズ: タスクの生成

**設計が検証に合格した場合 (GO 決定)**:
- レビューのフィードバックを確認し、必要に応じて変更を適用する
- `$tp-spec-tasks $1` を実行して実装タスクを生成する
- または `$tp-spec-tasks $1 -y` で自動承認し、直接次に進む

**設計に修正が必要な場合 (NO-GO 決定)**:
- 特定された重大な問題に対処する
- 改善を加えて `$tp-spec-design $1` を再実行する
- `$tp-validate-design $1` で再検証する

**注**: 設計のバリデーションは推奨されますが、任意です。品質レビューにより問題を早期に発見することができます。
