# 要件生成

<background_information>
- **成功基準**:
  - ステアリングコンテキストと整合した完全な要件ドキュメントを作成する
  - すべての受入基準に関して、プロジェクトの EARS パターンと制約に従う
  - 実装の細部ではなく、コアとなる機能に焦点を当てる
  - スコープが誤って解釈される可能性がある場合、包含/除外の境界を明示的にする
  - メタデータを更新して生成状況を追跡する
</background_information>

<instructions>
## 実行手順

1. **コンテキストの読み込み**:
   - 言語とメタデータについて `.exec-plans/specs/$1/spec.json` を読み込む
   - `.exec-plans/specs/$1/brief.md` が存在する場合は読み込む（ディスカバリーのコンテキスト：問題、アプローチ、範囲の決定、境界の候補）
   - プロジェクト説明について `.exec-plans/specs/$1/requirements.md` を読み込む
   - コアのステアリングコンテキスト: `product.md`, `tech.md`, `structure.md`
   - 追加のステアリングファイルは、機能スコープ、ユーザーペルソナ、ビジネス/ドメインルール、コンプライアンス/セキュリティ制約、運用上の制約、または既存の製品境界に直接関係する場合にのみ読み込む
   - 関連するローカルエージェントのスキルやプレイブックは、機能のホスト環境やユースケースに明確に一致し、ユーザーが観察可能な要件を形成する ドメイン用語やワークフロールール が含まれている場合にのみ読み込む

2. **ガイドラインの確認**:
   - EARS構文ルールのために、このスキルディレクトリの `rules/ears-format.md` を読み込む
   - 書き込み前レビュー基準のために、このスキルディレクトリの `rules/requirements-review-gate.md` を読み込む
   - ドキュメント構造のために、`.exec-plans/settings/templates/specs/requirements.md` を読み込む

#### 並行調査 (サブエージェントのディスパッチ)

以下の調査領域は独立しています。プロジェクトの複雑さに応じて最適な分解（サブエージェントの分割、統合、追加、またはスキップ）を決定してください。

**メインコンテキスト内** (要件生成に不可欠):
- 仕様ファイル: spec.json, brief.md, requirements.md (プロジェクトの説明)
- EARSフォーマットルール、要件レビューゲート、要件テンプレート
- コアのステアリング: product.md, tech.md (スコープや制約に関する直接の情報を与える)

**サブエージェントへの委任** (探索をメインコンテキストから分離する):
- **コードベースのヒント** (ブラウンフィールド（既存）プロジェクト): 要件範囲に影響を与える既存の実装を探索するサブエージェントを生成します。(1) すでに存在するもの、(2) 関連するインターフェース/API、(3) 新しい要件が合わせるべきパターン、を要約するように指示し、150行未満の要約を返させます。
- **ドメイン調査** (外部の知識が必要な場合): ドメイン固有の要件、標準、またはベストプラクティスについてウェブ調査を行うサブエージェントを生成します。簡潔な調査結果の要約を返させます。
- **追加のステアリングとプレイブック**: 多くのステアリングファイルやローカルエージェントのプレイブックが存在する場合、それらをスキャンしてこの機能に関連するセクションのみを返すサブエージェントを生成します。

コードベースが最小限のグリーンフィールド（新規）プロジェクトでは、サブエージェントのディスパッチをスキップし、コンテキストを直接ロードします。マルチエージェントが利用できない場合は、メインコンテキストで順次実行します。

すべての調査が完了したら、要件を生成する前にメインコンテキストで調査結果を統合します。

3. **要件ドラフトの生成**:
   - プロジェクトの説明に基づいて初期の要件ドラフトを作成する
   - 関連する機能を論理的な要件エリアにグループ化する
   - すべての受入基準に EARS フォーマットを適用する
   - spec.json で指定された言語を使用する
   - フェーズ間での用語の連続性を維持する:
     - discovery = `Boundary Candidates`
     - requirements = 必要な場合の明示的な包含/除外および隣接する期待事項(adjacent expectations)
     - design = `Boundary Commitments`
     - tasks = `_Boundary:_`
   - スコープが誤って解釈される可能性がある場合は、実装やアーキテクチャの所有権に関する詳細を持ち込まずに、軽量な境界のコンテキストを追加する
   - レビューゲートを通過するまでこれをドラフトとして保持し、`requirements.md` にまだ書き込まないこと

4. **要件ドラフトのレビュー**:
   - `rules/requirements-review-gate.md` から `Requirements Review Gate` を実行する
   - 決定前に、網羅性、EARSコンプライアンス、曖昧さ、隣接する期待事項、およびスコープの境界をレビューする
   - 問題がドラフト内で局所的な場合は、要件を修復して再度レビューする
   - レビューは最大2回の修復パスに制限する
   - ドラフトがスコープに関する本当の曖昧さや矛盾を明らかにした場合、推測で要件を書くのではなく、停止してユーザーに明確化を求める

5. **確定とメタデータの更新**:
   - 要件レビューゲートに合格した後にのみ `.exec-plans/specs/$1/requirements.md` を書き込む
   - `phase: "requirements-generated"` を設定する
   - `approvals.requirements.generated: true` を設定する
   - `updated_at` タイムスタンプを更新する

## 重要な制約

### 要件のスコープ: HOW ではなく WHAT
要件はユーザーが観察可能な挙動を説明するものであり、実装ではありません。これを基準にして、ここに記載すべきことと設計段階に委ねるものを決定してください:

**ユーザーに尋ねる項目（要件のスコープ）:**
- 機能的スコープ — 何が含まれ、何が除外されるか
- ユーザーが観察可能な挙動 — 「Xが発生したとき、ユーザーは何を見るべきか/経験すべきか？」
- ビジネスルールとエッジケース — 制限、エラー条件、特別なケース
- ユーザーに見える非機能要件 — 提供されるべき応答時間、可用性、セキュリティレベル
- 隣接する期待要件（ユーザーの目に見える挙動やオペレーターの期待事項を変える場合のみ） — この機能が依存するもの、および明示的に当機能が所有しないもの

**尋ねない項目（設計のスコープ — 設計フェーズに委ねる）:**
- 技術スタックの選択 (データベース、フレームワーク、言語)
- アーキテクチャパターン (マイクロサービス、モノリス、イベント駆動)
- API設計、データモデル、内部コンポーネント構造
- 非機能要件をどのようにして達成するか (キャッシュ戦略、スケーリングアプローチ)
- 内部の所有権マッピング、コンポーネントの継ぎ目、または設計に属する実装の境界

**リトマス試験紙**: EARSの受入基準がテクノロジーに言及せずに書き表せるなら、それは要件に属します。テクノロジーの選択を必要とするなら、それは設計に属します。

### その他の制約
- 各要件はテスト可能で曖昧さがないものでなければなりません。プロジェクトの説明でスコープ、振る舞い、または境界条件に関する複数の解釈の余地がある場合は、その要件を生成する前にユーザーに明確にするよう求めてください。必要なだけ質問し、独自の仮定を含む要件を生成しないでください。
- EARS ステートメントの適切な主語を選択してください（ソフトウェアのシステム/サービス名）
- requirements.md 内の要件の見出しには、必ず先頭の数値 ID のみを含める必要があります（例: "Requirement 1"、"1."、"2 Feature ..."）。"Requirement A"のようなアルファベットのIDは使用しないでください。
</instructions>

## 出力の説明
spec.json で指定された言語を使用して、以下の内容を含む出力を提供してください:

1. **生成された要件の要約**: 主要な要件分野の簡潔な概要（3〜5ポイント）
2. **ドキュメントのステータス**: requirements.md が更新され、spec.json のメタデータが更新されたことの確認
3. **レビューゲート**: 要件レビューゲートに合格したことの確認
4. **次のステップ**: 進め方に関するユーザーへのガイダンス（承認して続行するか、または変更するか）

**フォーマット要件**:
- 明確さのために Markdown の見出しを使用する
- コードブロック内にファイルパスを含める
- 要約は簡潔に保つ（300語未満）

## 安全性とフォールバック

### エラーシナリオ
- **プロジェクト説明の欠落**: requirements.md にプロジェクトの概要が欠けている場合、機能の詳細をユーザーに尋ねる
- **テンプレートの欠落**: テンプレートファイルが存在しない場合は、警告を出してインラインのフォールバック構造を使用する
- **言語未定義**: spec.json で言語が指定されていない場合は、デフォルトで英語 (`en`) となる
- **不完全な要件**: 生成後、期待されるすべての機能が要件でカバーされているかユーザーに明示的に確認する
- **空のステアリングディレクトリ**: プロジェクトコンテキストが不足しており、要件品質に影響を与える可能性があることをユーザーに警告する
- **数値でない要件の見出し**: 既存の見出しに先頭の数値IDが含まれていない場合（例えば "Requirement A" を使用している場合）、数値IDを使用するように正規化し、そのマッピングを一貫して保つこと（数値とアルファベットのラベルを決して混在させない）。
- **要件プレビュー中のスコープ定義の曖昧さ**: 要件レビュー中に欠落または競合するスコープが判明した場合は、推測での `requirements.md` 書き込みを止め、ユーザーに確認を求めてから `$tp-spec-requirements $1` を再実行する。

### 次のフェーズ: 設計の生成

**要件が承認された場合**:
- `.exec-plans/specs/$1/requirements.md` で生成された要件をレビューする
- **オプションのギャップ分析** (既存のコードベース向け):
  - `$tp-validate-gap $1` を実行し、現在のコードとの実装のギャップを探る
  - 既存のコンポーネント、統合ポイント、実装戦略を特定する
  - 既存プロジェクト（ブラウンフィールド）には推奨し、新規（グリーンフィールド）にはスキップする
- その後、`$tp-spec-design $1 -y` を実行して設計フェーズに進む

**変更が必要な場合**:
- フィードバックを提供し、`$tp-spec-requirements $1` を再実行する
