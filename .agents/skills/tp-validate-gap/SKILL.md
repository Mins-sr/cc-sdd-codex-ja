# 実装ギャップのバリデーション

<background_information>
- **ミッション**: 要件と既存のコードベースの間のギャップを分析し、実装戦略に情報を提供する
- **成功基準**:
  - 既存のコードベースのパターンとコンポーネントを包括的に理解していること
  - 不足している機能や統合の課題を明確に特定できていること
  - 実行可能な複数の実装アプローチが評価されていること
  - 設計フェーズで必要となる技術的調査のニーズが特定されていること
</background_information>

<instructions>
## コアタスク
承認された要件と既存のコードベースに基づいて、機能 **$1** の実装ギャップを分析します。

## 実行手順

1. **コンテキストの読み込み**:
   - 言語とメタデータについて `.exec-plans/specs/$1/spec.json` を読み込む
   - 要件について `.exec-plans/specs/$1/requirements.md` を読み込む
   - コアのステアリングコンテキスト: `product.md`, `tech.md`, `structure.md`
   - 追加のステアリングファイルは、機能のドメインルール、統合、ランタイムの前提条件、コンプライアンス/セキュリティ制約、または既存の製品境界に直接関係する場合にのみ読み込む
   - 関連するローカルエージェントのスキルやプレイブックは、機能のホスト環境やユースケースに明確に一致し、分析に関連するコンテキストを提供する場合にのみ読み込む

2. **分析ガイドラインの確認**:
   - 包括的な分析フレームワークのために、このスキルディレクトリの `rules/gap-analysis.md` を読み込む

#### 並列調査

以下の調査領域は独立しており、並行して実行できます:
1. **コードベース分析**: 既存の実装、アーキテクチャパターン、統合ポイント、拡張の可能性
2. **外部依存関係の調査**: 依存関係の互換性、バージョン制約、既知の統合課題（必要な場合）
3. **コンテキストの読み込み**: 要件、コアステアリング、タスク関連の追加ステアリング、関連するローカルエージェントスキル/プレイブック、および gap-analysis のルール

マルチエージェントが有効な場合、上記の各領域に対してサブエージェントを生成します。それ以外の場合は順次実行します。

すべての並行調査が完了した後、ギャップ分析のために調査結果を統合します。

3. **ギャップ分析の実行**:
   - 徹底的な調査のために gap-analysis.md のフレームワークに従う
   - Grep ツールと Read ツールを使用して既存のコードベースを分析する
   - 必要な場合は WebSearch/WebFetch を使用して外部依存関係を調査する
   - 複数の実装アプローチ（拡張/新規/ハイブリッド）を評価する
   - 出力には spec.json で指定された言語を使用する

4. **分析ドキュメントの生成**:
   - gap-analysis.md の出力ガイドラインに従い、包括的なギャップ分析を作成する
   - トレードオフを伴う実行可能な複数のオプションを提示する
   - さらなる調査が必要な領域にフラグを立てる

5. **ギャップ分析をディスクに書き込む**:

   **ギャップ分析をディスクに書き込み、セッション境界を越えて設計フェーズで参照できるようにします。**

   - ギャップ分析を `.exec-plans/specs/$1/research.md` に保存する
   - ファイルがすでに存在する場合は、以前の調査を上書きするのではなく、新しい分析を追記する（水平線 `---` で区切る）
   - ファイルを読み直して、正常に書き込まれたか確認する

## 重要な制約
- **決定よりも情報**: 最終的な実装の選択ではなく、分析とオプションを提供する
- **複数のオプション**: 該当する場合は、実行可能な代替案を提示する
- **徹底的な調査**: ツールを使用して既存のコードベースを深く理解する
- **明示的なギャップ**: 調査や検討が必要な領域を明確にフラグ付けする
- **コンテキストの規律**: コアステアリングから開始し、分析に関連するステアリングやユースケースに沿ったローカルエージェントのスキル/プレイブックのみを拡張する
</instructions>

## ツールガイダンス
- **最初に読む**: 分析の前に、仕様、コアステアリング、関連するローカルプレイブック/エージェントスキル、およびルールをロードする
- **広範な Grep**: パターン、規則、統合ポイントについてコードベースを検索する
- **WebSearch/WebFetch**: 必要な場合は外部の依存関係やベストプラクティスを調査する
- **最後に書く**: 完全な調査が終了した後にのみ分析を生成する

## 出力の説明
spec.json で指定された言語を使用して、以下の内容を含む出力を提供してください:

1. **分析の要約**: スコープ、課題、推奨事項の簡潔な概要（3〜5個の箇条書き）
2. **ドキュメントのステータス**: 使用された分析アプローチを確認する
3. **次のステップ**: ユーザーが設計フェーズに進むためのガイダンス

**フォーマット要件**:
- 明確さのために Markdown の見出しを使用する
- 要約は簡潔に保つ（300語未満）
- 詳細な分析は gap-analysis.md の出力ガイドラインに従う

## 安全性とフォールバック

### エラーシナリオ
- **要件の欠落**: requirements.md が存在しない場合は、メッセージ「要件を生成するために先に `$tp-spec-requirements $1` を実行してください」を表示して停止する
- **要件が承認されていない**: 要件が承認されていない場合はユーザーに警告するが、処理は続行する（ギャップ分析により要件の修正情報を提供できるため）
- **空のステアリングディレクトリ**: プロジェクトコンテキストが不足しており、分析品質に影響を与える可能性があることをユーザーに警告する
- **複雑な統合が不明確**: ブロックするのではなく、設計フェーズでの包括的な調査のためにフラグを立てる
- **言語未定義**: spec.json で言語が指定されていない場合は、デフォルトで英語 (`en`) となる

### 次のフェーズ: 設計の生成

**ギャップ分析が完了した場合**:
- ギャップ分析による洞察をレビューする
- `$tp-spec-design $1` を実行して技術設計ドキュメントを作成する
- または `$tp-spec-design $1 -y` で要件を自動承認し、直接次に進む

**注**: ギャップ分析は任意ですが、設計の決定に情報を提供するため、ブラウンフィールド（既存の）プロジェクトでは推奨されます。
