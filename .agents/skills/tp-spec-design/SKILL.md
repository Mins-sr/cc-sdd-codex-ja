# 技術設計ジェネレータ

<background_information>
- **成功基準**:
  - すべての要件が、明確なインターフェースを持つ技術コンポーネントにマッピングされている
  - タスクの生成とレビューの指針となるよう、責任の境界(boundaries)を十分に明確にしている
  - 適切なアーキテクチャのディスカバリー(発見)と調査が完了している
  - 設計がステアリングのコンテキストと既存のパターンと整合している
  - 複雑がアーキテクチャに対して視覚的な図(ダイアグラム)が含まれている
</background_information>

<instructions>
## 実行手順

### ステップ 1: コンテキストの読み込み

**必要なすべてのコンテキストを読む**:
- `.exec-plans/specs/$1/spec.json`, `requirements.md`, `design.md` (存在する場合)
- `.exec-plans/specs/$1/research.md` (存在する場合、`$tp-validate-gap` のギャップ分析を含んでいる)
- コアのステアリングコンテキスト: `product.md`, `tech.md`, `structure.md`
- 追加のステアリングファイルは、要件カバー率、アーキテクチャの境界、統合、ランタイムの前提条件、セキュリティ/パフォーマンスの制約、または実装の準備に影響を与えるチームの規約に直接関係する場合にのみ読み込む
- ドキュメント構造のために `.exec-plans/settings/templates/specs/design.md` を読み込む
- 設計原則のために、このスキルのディレクトリから `rules/design-principles.md` を読み込む
- 調査ログ構造のために `.exec-plans/settings/templates/specs/research.md` を読み込む

**要件承認の検証**:
- `-y` フラグが指定された場合 ($2 == "-y"): spec.json で requirements(要件)を自動承認する
- そうでない場合: 承認ステータスを検証する (未承認の場合は停止する、Safety & Fallback を参照)

### ステップ 2: ディスカバリーと分析 (Discovery & Analysis)

**重要: このフェーズは、設計が完全で正確な情報に基づいていることを保証します。**

1. **機能タイプの分類**:
   - **新機能 (New Feature)** (グリーンフィールド) → 完全な調査が必要
   - **拡張 (Extension)** (既存システム) → 統合を中心とした調査
   - **単純な追加 (Simple Addition)** (CRUD/UI) → 最小限の調査、または調査なし
   - **複雑な統合 (Complex Integration)** → 包括的な分析が必要

2. **適切な調査(ディスカバリー)プロセスの実行**:
   
   **複雑な機能 / 新機能の場合**:
   - このスキルのディレクトリから `rules/design-discovery-full.md` を読んで実行する
   - WebSearch/WebFetch を使用して徹底的な調査を実施する:
     - 最新のアーキテクチャパターンとベストプラクティス
     - 外部依存関係の検証 (API、ライブラリ、バージョン、互換性)
     - 公式ドキュメント、移行ガイド、既知の問題
     - パフォーマンスベンチマークとセキュリティの考慮事項
   
   **拡張の場合**:
   - このスキルのディレクトリから `rules/design-discovery-light.md` を読んで実行する
   - 統合ポイント、既存のパターン、互換性に焦点を当てる
   - Grep を使用して既存のコードベースのパターンを分析する
   
   **単純な追加の場合**:
   - 正式な調査を行わず、簡単なパターンのチェックのみを行う

#### 並行調査 (サブエージェントのディスパッチ)

以下の調査領域は独立しており、**サブエージェント**としてディスパッチできます。機能の複雑さに応じて、サブエージェントの分割、統合、追加、またはスキップするかどうか、最適な分解を決定してください。メインのコンテキストを統合（synthesis）用にクリーンに保つため、各サブエージェントは（生データではなく）**調査結果の要約 (findings summary)** を返します。

**典型的な調査領域** (必要に応じて調整):
- **コードベース分析**: 既存のアーキテクチャパターン、統合ポイント、コード規約
- **外部調査**: 依存関係、API、最新のベストプラクティス
- **コンテキストの読み込み** (通常はメインコンテキスト内): ステアリングファイル、設計原則、ディスカバリールール、テンプレート

単純な追加の場合、サブエージェントのディスパッチを完全にスキップし、メインコンテキストで簡単なパターンのチェックを行います。

すべての調査結果が戻った後、次のステップに進む前にメインコンテキストで統合します。

3. **ステップ3のためにディスカバリー結果を保持する**:
   - 外部APIの契約（コントラクト）と制約
   - 根拠を伴う技術的な決定
   - 従うべき、あるいは拡張すべき既存のパターン
   - 統合ポイントと依存関係
   - 特定されたリスクと軽減戦略
   - 境界の候補、境界外とするもの（out-of-boundary）の決定、そして可能性の高い再検証トリガー

4. **調査結果をリサーチログ（Research Log）に永続化する**:
   - 共有テンプレートを使用して `.exec-plans/specs/$1/research.md` を作成または更新する
   - ディスカバリーの範囲と主要な発見（Summary セクション）を要約する
   - "Research Log" のトピック（テーマ）での調査を、情報源および意味合い（implications）とともに記録する
   - テンプレートセクションを使用して、アーキテクチャパターンの評価、設計の決定、およびリスクを文書化する
   - `research.md` を書いたり更新したりする際は spec.json で指定されている言語を使用する

### ステップ 3: 統合 (Synthesis)

**書き込む前に、ディスカバリーの発見に設計の統合(synthesis)を適用します。**

- このスキルのディレクトリから `rules/design-synthesis.md` を読んで適用する
- このステップでは、ディスカバリーからの全体像が必要です。並行処理したりサブエージェントに委任したりしないでください。
- 統合の結果 (見つかった一般化、開発・構築(build)するか採用するか(adopt)の決定、単純化) を `research.md` に記録する

### ステップ 4: ドラフト（設計草案）の生成

1. **設計テンプレートとルールの読み込み**:
   - 構造について `.exec-plans/settings/templates/specs/design.md` を読み込む
   - 原則について、このスキルのディレクトリの `rules/design-principles.md` を読み込む

2. **設計ドラフトの生成**:
   - **`specs/design.md` テンプレート構造と生成指示に厳密に従う**
   - **境界ファーストの要件事項**: サポートするセクションを拡張する前に、境界を明示的にする。この仕様が何を所有し、何を所有「しない」のか、どの依存関係が許可されているか、どのような変更が加えられたら下流の再検証(revalidation)が必要になるかを、ドラフト内で明確に定義しなければならない。
   - **調査で分かったことすべてと統合の結果(synthesis outcomes)を統合する**: コンポーネントの定義、アーキテクチャの決定、および統合のポイントを通じて、調査情報 (API、パターン、テクノロジー) と統合の決定事項 (一般化、開発-vs-採用、単純化) を使用する
   - **ファイル構造計画 (必須)**: ファイル構造計画セクションに、具体的なファイルパスとそれぞれの責任を入力する。コードベースを分析して、作成対象と変更対象を決定する。各ファイルには明確な責任が1つなければならない。このセクションはタスクの `_Boundary:_` アノテーションと実装時のタスクブリーフを直接駆動させる。曖昧なファイル構造は曖昧な実装を生み出す。
   - **テスト戦略**: 一般的なパターンからではなく、要件の受入基準からテスト項目を導き出す。「ログインが動作することをテストする」のようなあいまいな記載は避け、代わりに何が検証されていてなぜそれが重要なのかを正確に指定する。
   - ステップ1で既存の design.md が見つかった場合は、それを参照コンテキストとして使用する (マージモード)
   - 設計ルール（「型安全性 (Type Safety)」「視覚的コミュニケーション (Visual Communication)」「フォーマルなトーン (Formal Tone)」）を適用する
   - spec.json で指定された言語を使用する
   - レビューゲートに合格するまでこれをドラフトとして保持すること。まだ `design.md` に書き込まない

### ステップ 5: 設計ドラフトのレビュー

- このスキルのディレクトリから `rules/design-review-gate.md` を読み込み適用する
- 設計を完了する前に、要件のカバー率、アーキテクチャの準備状況、および実装の実行可能性を検証する
- もしドラフト内の局所的な問題であれば、設計を修正して再度レビューする
- レビューは最大2回の調整(修復)パスに制限する
- もしドラフトから実際の「要件/設計」のギャップが明らかになった場合、`design.md` でその場を取り繕うのではなく、停止して要件の明確化の段階へ戻ること

### ステップ 6: 設計ドキュメントの完了確定（ファイナライズ）

1. **最終的な設計とリサーチログの書き込み**:
   - 設計のレビューゲートをに合格した後にのみ、`.exec-plans/specs/$1/design.md` を書き込む
   - ディスカバリーでの調査結果および統合(synthesis)の決定結果を `research.md` に書き込む (まだ書き込まれていない場合)
   - 完成した設計をサポートする `research.md` の更新内容があればそれを永続化する

2. **spec.json のメタデータを更新**:
   - `phase: "design-generated"` を設定する
   - `approvals.design.generated: true, approved: false` を設定する
   - `approvals.requirements.approved: true` を設定する
   - `updated_at` タイムスタンプを更新する

## 重要な制約
 - **型の安全性 (Type Safety)**:
   - プロジェクトのテクノロジースタックに合わせた強力な型付けを強制します。
   - 静的型付け言語の場合、明示的な型/インターフェースを定義し、安全でないキャストを避けます。
   - TypeScript の場合、`any` は絶対に使用せず、正確な型とジェネリクスを優先します。
   - 動的型付け言語の場合、利用可能な場所（例：Python の type hints）で型のヒント/注釈を提供し、境界で入力を検証します。
   - コンポーネント間の型の安全性を確保するため、公開インターフェースやコントラクトを明確に文書化します。
- **要件追跡 ID (Requirements Traceability IDs)**: requirements.md の定義と完全に一致する数値のみの要件 ID (例: "1.1", "1.2", "3.1", "3.3") を使用してください。新しい識別子(ID)を勝手に作ったり、アルファベットを使用したラベルを用いたりしないでください。
</instructions>

## 出力の説明

**コマンド実行の出力** (`design.md` の内容とは別で):

spec.json で指定されている言語を利用して簡単な要約を提供する。

1. **ステータス**: 設計ドキュメントが `.exec-plans/specs/$1/design.md` に生成されたことの確認
2. **ディスカバリー(調査)のタイプ**: どのプロセスで実行されたかの確認 (full / light / minimal)
3. **キーとなる発見**: 設計を形作った、`research.md` からの2〜3個の重要な洞察
4. **レビューゲート**: 設計のレビューゲートを合格したかどうかの確認
5. **次のアクション**: 承認ワークフローのガイダンス (Safety & Fallback を参照)
6. **リサーチログ**: `research.md` が最新の決定事項で更新されたことの確認

**フォーマット**: 簡潔な Markdown 形式 (200語以内) - これは設計ドキュメントそのものではなく、コマンド出力。

**注**: 実際の設計ドキュメントは `.exec-plans/settings/templates/specs/design.md` 構造に基づきます。

## 安全性とフォールバック

### エラーシナリオ

**要件が承認されていない**:
- **実行の停止**: 承認された要件なしには進めません
- **ユーザーへのメッセージ**: "要件がまだ承認されていません。設計生成の前に承認が必要です。"
- **推奨アクション**: "$tp-spec-design $1 -y を実行して要件を自動承認して進む"

**要件の欠落**:
- **実行の停止**: requirements.md が存在している必要があります
- **ユーザーへのメッセージ**: "`.exec-plans/specs/$1/requirements.md` に要求仕様が見つかりません"
- **推奨アクション**: "まずは `$tp-spec-requirements $1` を実行して、要件を生成してください"

**テンプレートの欠落**:
- **ユーザーへのメッセージ**: "`.exec-plans/settings/templates/specs/design.md` にテンプレートファイルがありません。"
- **推奨アクション**: "リポジトリのセットアップを確認するか、テンプレートファイルを復旧してください。"
- **フォールバック**: 警告とともに基本的なインライン構造を使用する。

**ステアリングコンテキストの欠落**:
- **警告**: "ステアリング(Steering)ディレクトリが空または欠落しています - 設計がプロジェクト標準に沿っていない可能性があります"
- **処置**: 生成は続行するが、出力で制約を明記する。

**無効な要件 ID**:
  - **実行停止**: もし requirements.md に数値の ID が見当たらないか、数値以外の見出し（例えば "Requirement A"）が使われている場合、実行を停止し、ユーザに requirements.md を修復するように促してください。

**設計のレビュー中に仕様のギャップが見つかった場合**:
- **実行の停止**: その場を取り繕った `design.md` を書き込まない
- **ユーザーへのメッセージ**: "設計レビュー中に、設計を確定させる前に解決すべき実際の仕様のギャップや曖昧さが見つかりました。"
- **推奨アクション**: `requirements.md`を明確にするか修正した後、再度 `$tp-spec-design $1` を実行してください

### 次のフェーズ: タスク生成

**設計が承認された場合**:
- `.exec-plans/specs/$1/design.md` で生成された設計をレビューする
- **オプション**: インタラクティブな品質レビューのために `$tp-validate-design $1` を実行する
- その後 `$tp-spec-tasks $1 -y` を実行して実装タスクを生成する

**変更が必要な場合**:
- フィードバックを提供し、`$tp-spec-design $1` を再実行する
- 既存の設計が参照 (マージモード) として使用される
