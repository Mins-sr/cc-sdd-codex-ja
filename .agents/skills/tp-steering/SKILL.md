# Tp ステアリング管理

<background_information>
**役割**: `.exec-plans/steering/` を永続的なプロジェクトメモリとして維持すること。

**ミッション**:
- ブートストラップ（初期化）: コードベースからコアとなるステアリングを生成する（初回）
- 同期 (Sync): ステアリングとコードベースを一致させる（保守）
- 保持: ユーザーによるカスタマイズは不可侵であり、更新は追記的（additive）であること

**成功基準**:
- ステアリングが、網羅的なリストではなく、パターンと原則を捉えていること
- コードベースのドリフト（逸脱）が検出され、報告されること
- `.exec-plans/steering/*.md` のすべて（コアとカスタム）が同等に扱われること
</background_information>

<instructions>
## シナリオ検出

`.exec-plans/steering/` のステータスを確認します:

**ブートストラップ モード**: 空であるか、コアファイル（product.md, tech.md, structure.md）が欠落している場合
**同期 (Sync) モード**: すべてのコアファイルが存在する場合

---

## ブートストラップ フロー

1. `.exec-plans/settings/templates/steering/` からテンプレートを読み込む
2. コードベースを分析する (JIT):

#### 並列調査

以下の調査領域は独立しており、並行して実行できます:
1. **製品分析**: README、package.json、ドキュメントファイルから、目的、価値、コア機能を分析
2. **技術分析**: 設定ファイル、依存関係、フレームワークからテクノロジーのパターンと決定を分析
3. **構造分析**: ディレクトリツリー、命名規則、インポートパターンから組織構成を分析

マルチエージェントが有効な場合、上記の各領域に対してサブエージェントを生成します。それ以外の場合は順次実行します。

すべての並行調査が完了した後、ステアリングファイルに向けてパターンを統合します。

3. パターン（リストではない）を抽出する:
   - プロジェクト (Product): 目的、価値、コアとなる機能
   - テック (Tech): フレームワーク、決定事項、規則
   - 構造 (Structure): 組織構成、ネーミング、インポート
4. ステアリングファイルを生成する（テンプレートに従う）
5. このスキルディレクトリの `rules/steering-principles.md` から原則を読み込む
6. レビュー用の要約を提示する

**焦点**: ファイルや依存関係のカタログではなく、決定を導く「パターン」に焦点を当てる。

---

## 同期 (Sync) フロー

1. 既存のすべてのステアリング (`.exec-plans/steering/*.md`) を読み込む
2. 変更についてコードベースを分析する (JIT)
3. ドリフト（逸脱）を検出する:
   - **ステアリング → コード**: 要素の欠落 → 警告
   - **コード → ステアリング**: 新しいパターン → 更新候補
   - **カスタムファイル**: 関連性のチェック
4. 更新を提案する（追記的であり、ユーザーコンテンツを保持すること）
5. 報告する: 更新、警告、推奨事項

**アップデートの哲学**: 置き換えではなく、追加すること。ユーザーのセクションを保持すること。

---

## 粒度の原則

`rules/steering-principles.md`（このスキルのディレクトリ内）から：

> 「新しいコードが既存のパターンに従っている場合、ステアリングを更新する必要はない。」

網羅的なリストではなく、パターンと原則を文書化してください。

**悪い例**: ディレクトリツリー内のすべてのファイルをリストアップする
**良い例**: 構成パターンを具体例で記述する

</instructions>

## ツールガイダンス

- **Glob**: ソースファイルや設定ファイルを探す
- **Read**: ステアリング、仕様書、設定を読む
- **Grep**: パターンを検索する
- **Bash** (`ls`を使用): 構造を理解する

**JIT 戦略**: 必要なときに取得し、最初からすべてを読み込まない。

## 出力の説明

チャットでの要約のみ (ファイルは直接更新されます)。

### ブートストラップ時:
```
✅ ステアリングが作成されました

## 生成されたもの:
- product.md: [簡潔な説明]
- tech.md: [主要な技術スタック]
- structure.md: [組織構成]

確認し、信頼できる情報源（Source of Truth）として承認してください。
```

### 同期 (Sync) 時:
```
✅ ステアリングが更新されました

## 変更内容:
- tech.md: React 18 → 19 に変更
- structure.md: API パターンが追加されました

## コードからの逸脱 (Drift):
- インポートの規則に従っていないコンポーネントがあります

## 推奨事項:
- api-standards.md の作成を検討してください
```

## 例

### ブートストラップ
**入力**: 空のステアリング、React TypeScript のプロジェクト
**出力**: パターンを含む3つのファイル - 「機能ファースト」「TypeScript strict」「React 19」

### 同期 (Sync)
**入力**: 既存のステアリング、新しい `/api` ディレクトリが存在する
**出力**: 更新された structure.md、非準拠ファイルへのフラグ付け、api-standards.md の作成の提案

## 安全性とフォールバック

- **セキュリティ**: キー、パスワード、秘密情報を含めない（原則を参照）
- **不確実性**: 両方の状態を報告し、ユーザーに尋ねる
- **保持の優先**: 迷ったときは、置き換えるのではなく「追加」する

## ノート

- すべての `.exec-plans/steering/*.md` ファイルはプロジェクトメモリとして読み込まれる
- カスタマイズのために、テンプレートと原則は外部の別ファイルとする
- カタログではなく、パターンに焦点を当てる
- 「ゴールデンルール（黄金律）」：パターンに従った新しいコードは、ステアリングの更新を必要としてはならない
- エージェント固有のツールディレクトリ (例: `.cursor/`, `.gemini/`, `.claude/`) の文書化は避ける
- `.exec-plans/settings/` のコンテンツをステアリングファイルに絶対に文書化しないこと（設定はメタデータであり、プロジェクトの知識ではない）
- `.exec-plans/specs/` および `.exec-plans/steering/` への軽い参照は許容される。他の `.exec-plans/` ディレクトリへの言及は避ける
