# 仕様のステータス

<background_information>
- **成功基準**:
  - 現在のフェーズと完了ステータスを表示する
  - 次のアクションとブロッカーを特定する
  - 進捗状況を明確に可視化する
  - 境界の準備状況、アップストリーム/ダウンストリームのコンテキスト、および再バリデーションが必要になりそうな項目がある場合はそれを表面化する
</background_information>

<instructions>
## 実行手順

### ステップ 1: 仕様コンテキストの読み込み
- `.exec-plans/specs/$1/spec.json` を読み込み、メタデータとフェーズのステータスを確認します
- `.exec-plans/specs/$1/brief.md` が存在する場合は読み込みます
- 既存のファイル: `requirements.md`, `design.md`, `tasks.md` が存在する場合は読み込みます
- 利用可能なファイルについて `.exec-plans/specs/$1/` ディレクトリを確認します
- `.exec-plans/steering/roadmap.md` が存在し、そこにこの仕様が記載されている場合は読み込みます

### ステップ 2: ステータスの分析

**各フェーズの解析**:
- **要件**: 要件と受入基準をカウントします
- **設計**: アーキテクチャ、コンポーネント、図解を確認し、境界セクションが存在するかを確認します
- **タスク**: 完了済みタスクと全タスクの比率をカウントします (`- [x]` と `- [ ]` を解析します)
- **承認**: spec.json の承認ステータスを確認します
- **境界コンテキスト**:
  - brief.md から: 存在する場合は `Boundary Candidates`, `Upstream / Downstream`, および `Existing Spec Touchpoints` を記録します
  - design.md から: 存在する場合は `Boundary Commitments`, `Out of Boundary`, `Allowed Dependencies`, および `Revalidation Triggers` を記録します
  - roadmap.md から: 上流の依存関係と、この仕様が `Existing Spec Updates` に隣接しているかどうかを記録します
- **再バリデーションウォッチリスト**:
  - この仕様が変更された場合に再バリデーションが必要になる可能性のある下流の仕様、隣接する既存仕様の更新、またはロールアウトに敏感な設計上の注意事項を特定します
  - 現在の仕様の形が広範すぎると見られ、より局所的な修復よりもロードマップ/設計の分割が望ましい可能性がある場合は、それを指摘します

### ステップ 3: レポートの生成

以下の項目をカバーするレポートを spec.json で指定された言語で作成します:
1. **現在のフェーズと進捗**: ワークフローにおける仕様の現在の位置
2. **完了ステータス**: 各フェーズの完了率（パーセンテージ）
3. **タスクのブレイクダウン**: タスクが存在する場合、完了済みと残りの中身を表示
4. **境界コンテキスト**: 該当する場合は、上流/下流、境界外、および許可された依存関係に関する注意事項
5. **再バリデーションウォッチリスト**: この仕様の変更によって影響を受ける可能性のある下流または隣接する作業
6. **次のアクション**: 次に実行する必要があること
7. **ブロッカー**: 進捗を妨げているあらゆる問題

</instructions>

## 安全性とフォールバック

### エラーシナリオ

**仕様が見つからない場合**:
- **メッセージ**: "No spec found for `$1`. Check available specs in `.exec-plans/specs/`"
- **アクション**: 利用可能な仕様ディレクトリをリスト化します

**不完全な仕様**:
- **警告**: どのファイルが欠落しているかを特定します
- **推奨されるアクション**: 次のフェーズのコマンドを案内します

### すべての仕様のリスト
利用可能なすべての仕様を確認するには:
- 引数なしで実行するか、ワイルドカードを使用します
- `.exec-plans/specs/` 内のすべての仕様とそのステータスを表示します
