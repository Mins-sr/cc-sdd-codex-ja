# 完了の検証 (tp-verify-completion)

<background_information>
このスキルは、誤った完了の主張を防ぎます。タスク、修正、または機能は、主張の範囲に一致する最新の証拠によって裏付けられた場合にのみ完了したとみなされます。
</background_information>

<instructions>
## いつ使用するか

- タスクが完了したと言う前に
- バグが修正されたと言う前に
- テストに合格したと言う前に
- 自律実行において次のタスクに移行する前に
- 機能レベルの検証から `GO` を報告する前に
- 別のサブエージェントの成功報告を信じる前に

早期の計画や推測によるステータス更新にこのスキルを使用しないでください。

## 入力

以下を提供してください:
- 検証する正確な主張
- 主張のタイプ:
  - `TASK`
  - `FIX`
  - `TEST_OR_BUILD`
  - `FEATURE_GO`
- コントローラーによって発見されたバリデーションのコマンド
- 最新のコマンド出力と終了コード (exit codes)
- 必要に応じて、関連するタスク ID、要件 ID、および設計の参照先
- 機能レベルの主張の場合：
  - 要件の網羅(カバレッジ)ステータス
  - 設計との整合性ステータス
  - 統合ステータス
  - ブロックされているタスクのステータス

## 出力

以下のいずれかを返します:
- `VERIFIED` (検証済み)
- `NOT_VERIFIED` (未検証)
- `MANUAL_VERIFY_REQUIRED` (手動検証が必要)

また、以下も返します:
- レビューされた主張
- 使用された証拠
- 範囲と証拠の不一致（存在する場合）

`spec.json` で指定された言語を使用してください。

## ゲート機能

1. 正確な主張を特定します。
2. その主張を証明する正確なコマンドまたはチェックリストを特定します。
3. コードの現在の状態からの最新の証拠を要求します。
4. 終了コード、失敗の数、スキップされたスコープ、およびカバレッジの不足をチェックします。
5. 証拠よりも広範な主張を拒否します。
6. 必須の検証を完了できない場合は、`MANUAL_VERIFY_REQUIRED` を返します。
7. その後にのみ、主張を許可します。

## 主張ごとの固有のルール

### TASK
必須要件:
- タスクにローカルな検証の証拠
- レビューによる未解決のブロッキングな指摘事項がないこと
- タスクの境界と証拠が一致していること

### FIX
必須要件:
- 元の症状が解決されたという証拠
- 関連する検証スコープに広範なリグレッションがないこと

### TEST_OR_BUILD
必須要件:
- 実際のコマンド出力
- 終了コード
- 無関係なチェック結果からの推論がないこと

### FEATURE_GO
必須要件:
- テストスイート全体の結果
- ビルドされたアーティファクトが最初の使用可能な状態に達したことを示す、ランタイムのスモークブートの結果
- 要件カバー率の評価
- クロスタスクの統合評価
- 設計のE2E (End-to-End) での整合性評価
- ブロックされているタスクの評価

テストスイートの合格だけでは `FEATURE_GO` には不十分です。

## 停止 / エスカレーション

以下の場合、`MANUAL_VERIFY_REQUIRED` を返します:
- 標準的なバリデーションコマンドが不明である
- 必要な環境が利用できない
- 必須の手動検証ステップが実行できない

以下の場合、`NOT_VERIFIED` を返します:
- コマンドが失敗した
- 証拠が古い(stale)
- 証拠が部分的である
- 主張が証拠の範囲を超えている
- 機能において、未解決のブロックされたタスクや、網羅されていない要件がまだ存在する

## よくある合理化（言い訳）

| 合理化 (言い訳) | 現実 |
|---|---|
| 「サブエージェントが成功したと言った」 | 報告された成功は検証の証拠ではありません。 |
| 「以前はテストに合格していた」 | 最新の証拠のみが有効です。 |
| 「リンターに合格したのでビルドは大丈夫なはずだ」 | lint はビルドの成功を証明しません。 |
| 「テストに合格しビルドも成功したので、実行できるに違いない」 | 型の消去、モジュールの読み込み、ネイティブABI、起動時の設定の問題により、ランタイムで失敗する可能性があります。 |
| 「すべてのタスクにチェックが入っているので機能は完了している」 | `FEATURE_GO` にはカバレッジ、統合、および設計の整合性も必要です。 |

## 出力フォーマット

```md
## 検証結果
- STATUS: VERIFIED | NOT_VERIFIED | MANUAL_VERIFY_REQUIRED
- CLAIM_TYPE: TASK | FIX | TEST_OR_BUILD | FEATURE_GO
- CLAIM: <正確な主張>
- EVIDENCE: <コマンド/チェックリストと結果>
- GAPS: <範囲/証拠の不一致、または不足している検証>
- NOTES: <検証されなかった場合の次のアクション>
```
</instructions>
