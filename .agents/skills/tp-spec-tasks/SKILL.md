# 実装タスクジェネレータ

<background_information>
- **成功基準**:
  - すべての要件が特定のタスクにマッピングされている
  - タスクのサイズが適切（各1〜3時間）である
  - 適切な階層構造を持ち、明確なタスクの進行になっている
  - 機能(capabilities)に焦点を当てた自然言語での説明
  - 軽量なタスク計画のサニティ(健全性)レビューにより、`tasks.md` が書き込まれる「前」にタスクグラフが実行可能であることを確認している
</background_information>

<instructions>
## 実行手順

### ステップ 1: コンテキストの読み込み

**必要なすべてのコンテキストを読む**:
- `.exec-plans/specs/$1/spec.json`, `requirements.md`, `design.md`
- `.exec-plans/specs/$1/tasks.md` (マージモードの場合、存在すれば)
- コアステアリングコンテキスト: `product.md`, `tech.md`, `structure.md`
- 要件カバレッジ、設計の境界、ランタイムの前提条件、またはタスクの実行可能性に影響を与えるチームの規約に直接関係する場合にのみ、追加のステアリングファイルを読む

**承認のバリデーション**:
- `-y` フラグが指定されている場合: spec.json 内の requirements(要件)と design(設計)を自動承認する。タスクの承認もステップ4で自動的に処理される。
- それ以外の場合: 両方が承認されているか確認する（されていない場合は停止する。安全性とフォールバックを参照）
- `--sequential` の有無に基づいてシーケンシャル（順次）モードを判定する

### ステップ 2: 実装タスクの生成

**生成ルールとテンプレートの読み込み**:
- 原則について、このスキルディレクトリから `rules/tasks-generation.md` を読む
- `sequential` が false の場合: 並行処理の判断基準について、このスキルディレクトリから `rules/tasks-parallel-analysis.md` を読む
- フォーマットについて `.exec-plans/settings/templates/specs/tasks.md` を読む (`(P)` マーカーをサポート)

#### 並行調査 (Parallel Research)

以下の調査領域は独立しており、並行して実行できる:
1. **コンテキストの読み込み**: 仕様ドキュメント(requirements.md, design.md)、ステアリングファイル
2. **ルールの読み込み**: tasks-generation.md, tasks-parallel-analysis.md, タスクテンプレート

マルチエージェントが有効な場合、上記の各領域に対してサブエージェントを生成する。それ以外の場合は順次実行する。

すべての並行調査が完了した後、タスクを生成する前に調査結果を統合する。

**すべてのルールに従ってタスクリストを生成する**:
- spec.json で指定された言語を使用する
- すべての要件をタスクにマッピングする
- 要件カバー(網羅)率を文書化する場合は、説明的な接尾辞、括弧、翻訳、または自由形式のラベルを付けず、数字の要件IDのみを（カンマ区切りで）リストアップする
- すべての設計コンポーネントが含まれていることを確認する
- タスクの進行が論理的かつ段階的（インクリメンタル）であることを検証する
- 実行可能な各サブタスクには、「完了」が観察可能な用語でどのようになるかを示す詳細な箇条書き(bullet)が少なくとも1つ含まれていることを確実にする
- 通常の実装タスクは単一の責任境界内に収める。作業が境界を越える場合は、それを明示的な統合タスクとする
- 単一のサブタスク構造を主要なタスクに昇格させることで折りたたみ、コンテナのみである主要なタスクに詳細を重複して記載することは避ける（必要に応じてテンプレートパターンを使用する）
- 並行処理の基準を満たすタスクに `(P)` マーカーを適用する（シーケンシャルモードが要求された場合はマーカーを省略する）
- テストカバレッジのオプション（任意）のサブタスクに `- [ ]*` をマークするのは、それが 中核となる実装によってすでに満たされている受け入れ基準を厳密にカバーしており、MVP後まで先送りできる場合 のみとする
- 既存の tasks.md が見つかった場合は、新しいコンテンツとマージする

### ステップ 3: タスク計画のレビュー

- ドラフトのタスク計画をワーキングメモリに保持する。まだ `tasks.md` を書き込まない
- `rules/tasks-generation.md` から「Task Plan Review Gate (タスク計画レビューゲート)」を実行する
- カバレッジのレビュー:
  - すべての要件IDが少なくとも1つのタスクに現れているか
  - すべての設計コンポーネント、契約 (contract)、統合ポイント、ランタイムの前提条件、および検証の関心事が網羅されているか
- 実行可能性のレビュー:
  - 各サブタスクは1〜3時間の実行可能な作業単位であるか
  - 各サブタスクには検証可能な成果物があるか
  - 実行可能な各サブタスクに、観察可能な完了を示す箇条書きが含まれているか
  - 隠れた暗黙の前提条件が残っていないか
  - `_Depends:_`、`_Boundary:_` および `(P)` マーカーが、引き続き依存関係グラフとアーキテクチャの境界に一致しているか
- 問題がタスク計画内部（ローカル）にある場合は、ドラフトを修復し、書き込む前にレビューゲートを再度実行する
- レビューは最大2回の修復パスに制限する
- 要件/設計のギャップや矛盾が明らかになった場合、水増しタスク(filler tasks)を発明するのではなく、停止してユーザーを要件/設計に戻す(修正させる)

### ステップ 3.5: タスクグラフのサニティ(健全性)レビューの実行

`tasks.md` を書き込む前に、タスクグラフに関する軽量で独立したサニティレビューを1回実行する。

- フレッシュなサブエージェントのディスパッチが利用可能な場合は、このステップのためにフレッシュなレビューサブエージェントを1つ生成する。そうでない場合は、現在のコンテキストで同じレビューを実行する。
- 既存の `tasks.md` が更新される場合は、ファイルパス、タスク計画ドラフト、マージコンテキストのみを提供する。レビューアは、親が統合したカバレッジサマリーに依存するのではなく、直接 `requirements.md`、`design.md`、およびタスク生成ルールを読み込むべきである。
- 以下の項目のみをチェックする:
  - 隠れた前提条件や欠落したセットアップタスク
  - 依存関係または順序付けのミス
  - タスク間の境界の重複または曖昧な所有権（責任）
  - タスクが大きすぎる、曖昧すぎる、明示的な統合タスクなしに境界をまたいでいる、または検証可能な成果物がない
  - 要件、設計、タスクグラフ間で矛盾が導入されていないか
- 以下の判定(verdict)のいずれかを返す:
  - `PASS`
  - `NEEDS_FIXES`
  - `RETURN_TO_DESIGN`
- `NEEDS_FIXES` の場合、ドラフトを1回修正し、サニティレビューを1回再実行する。
- `RETURN_TO_DESIGN` の場合、`tasks.md` を書き込まずに停止し、要件/設計の正確なギャップを指摘する。
- このフローは制限する。これを2回目の完全な計画サイクルにしないでください。

### ステップ 4: 完了確定（ファイナライズ）

**tasks.md を書き込む**:
- `.exec-plans/specs/$1/tasks.md` を作成/更新する
- spec.json のメタデータを更新する:
  - `phase: "tasks-generated"` を設定する
  - `approvals.tasks.generated: true, approved: false` を設定する
  - `approvals.requirements.approved: true` を設定する
  - `approvals.design.approved: true` を設定する
  - `updated_at` タイムスタンプを更新する

**承認プロセス**:
- 自動承認フラグ (`-y`) が指定されている場合:
  - spec.json に `approvals.tasks.approved: true` を設定する
  - タスクサマリー（タスク数、主要なグループ、並列マーカー）を表示する
  - 返答: "Tasks generated and auto-approved. Start implementation with `$tp-impl $1`" ("タスクが生成され、自動承認されました。`$tp-impl $1` で実装を開始してください")
- そうでない場合（対話式）:
  - 生成されたタスクのサマリー（タスク数、主要なグループ、並列マーカー）を表示する
  - ユーザーに尋ねる: "Tasks generated. Approve and proceed to implementation?" ("タスクの生成が完了しました。承認して実装に進みますか？")
  - ユーザーが承認した場合:
    - spec.json に `approvals.tasks.approved: true` を設定する
    - 返答: "Tasks approved. Start implementation with `$tp-impl $1`" ("タスクが承認されました。`$tp-impl $1` で実装を開始してください。")
  - ユーザーが変更を要求した場合:
    - `approvals.tasks.approved: false` のままにする
    - 何を調整して再実行すべきかについてのガイダンスを返答する

## 重要な制約
- **タスクの統合**: すべてのタスクはシステムに接続しなければならない（孤立した作業(orphaned work)を含めない）
- **境界アノテーション**: `(P)` タスクには必須であり、すべてに推奨される（例:`_Boundary: ComponentName_`）
- **明示的な依存関係**: 境界を越える自明でない依存関係は、例として `_Depends: X.X_` と宣言する
- **実行可能な成果物の粒度**: 各タスクは検証可能な成果物（ファイル、エンドポイント、UIコンポーネント、設定）を生成しなければならない。インフラストラクチャに関するタスク（プロジェクトの足場(スキャフォールディング)、マニフェスト、ホスト統合、ビルド構成）は明示的でなければならず、存在していると決して想定してはならない
- **観察可能な完了（done）のステータス**: 各実行可能サブタスクには、新しい事務管理用フィールドを追加することなく完了状態を可視化する、機能(details)の箇条書きを少なくとも1つ含める必要がある
- **暗黙の前提条件の禁止**: タスクがランタイム、SDK、フレームワークのセットアップ、または構成ファイルを必要とする場合、そのセットアップは（事前に）別の独立したタスクとしなければならない
</instructions>

## 出力の説明

spec.json で指定された言語で短いサマリーを提供してください:

1. **ステータス**: タスクが `.exec-plans/specs/$1/tasks.md` に生成されたことを確認する
2. **タスクのサマリー**: 
   - 合計: X 個の主要なタスク、Y 個のサブタスク
   - Z 個のすべての要件をカバー
   - 平均タスクサイズ: サブタスクごとに1〜3時間
3. **品質バリデーション**:
   - ✅ すべての要件がタスクにマッピングされた
   - ✅ 設計カバレッジとランタイム要件がレビューされた
   - ✅ タスク依存関係が検証された
   - ✅ タスク計画レビューゲートを通過した
   - ✅ 独立したタスクグラフの健全性(サニティ)レビューに合格した
   - ✅ テストタスクが含まれている
4. **次のアクション**: タスクを確認し、準備ができたら進む

**フォーマット**: 簡潔に（200語未満）

## 安全性とフォールバック

### エラーシナリオ

**要件または設計が承認されていない**:
- **実行の停止**: 承認された要件と設計がないと続行できない
- **ユーザーへのメッセージ**: "要件と設計はタスク生成前に承認されている必要があります"
- **推奨アクション**: "両方を自動承認して進める場合は、`$tp-spec-tasks $1 -y` を実行してください"

**要件または設計の欠落**:
- **実行の停止**: 両方の文書が存在しなければならない
- **ユーザーへのメッセージ**: "`.exec-plans/specs/$1/` に requirements.md または design.md が見つかりません"
- **推奨アクション**: "先に要件(requirements)と設計(design)のフェーズを完了してください"

**不完全な要件のカバー率(カバレッジ)**:
- **警告**: "すべての要件がタスクにマッピングされているわけではありません。カバレッジを確認してください。"
- **必要なユーザー対応**: 意図的なギャップであるか確認するか、タスクを再生成する

**タスクレビュー中に見つかった仕様のギャップ**:
- **実行の停止**: その場を取り繕った `tasks.md` を書き込まない
- **ユーザーへのメッセージ**: "要件/設計には、実行可能なタスク計画を生成するための十分で明確なカバレッジがありません"
- **推奨アクション**: "requirements.md または design.md を改善し、再度 `$tp-spec-tasks $1` を実行してください"

**テンプレート/ルールの欠落**:
- **ユーザーへのメッセージ**: "`.exec-plans/settings/` にテンプレートまたはルールファイルがありません"
- **フォールバック**: 警告付きでインラインの基本構造を使用する
- **推奨アクション**: "リポジトリのセットアップを確認するか、テンプレートファイルを復元してください"
- **要件の数値IDの欠落**:
  - **実行停止**: requirements.md内のすべての要件には数値IDが必要です。数値IDを持たない要件がある場合、タスクを生成する前に停止し、requirements.mdを修正するように要求してください。

### 次のフェーズ: 実装

タスクはステップ4でユーザーの確認によって承認されます。承認されたら:
- 自律的な実装: `$tp-impl $1`
- 特定のタスクのみ: `$tp-impl $1 1.1,1.2`
