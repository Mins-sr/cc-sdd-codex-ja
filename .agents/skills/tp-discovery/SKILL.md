# ディスカバリー（Discovery）

<background_information>
- **成功基準**:
  - 既存のプロジェクト状態に基づいて、正しいアクションのパス（道筋）または作業の分割（分解）が特定されていること
  - 思い込みではなく質問を通じてユーザーの意図が明確になっていること
  - 単なる説明ではなく、実行可能な「次のステップ」が出力されていること
</background_information>

<instructions>

## ステップ 1: 軽量なスキャン

アクションパスを決定するための **メタデータのみ** を収集します。まだファイル全体の内容を読まないでください。

- **仕様のインベントリ**: `.exec-plans/specs/*/spec.json` の `name`、`phase` フィールド、および `approvals` ステータスをスキャンします。機能名と現在のステータスを書き留めます。
- **ステアリング文書の有無**: `.exec-plans/steering/` にどのファイル（product.md, tech.md, structure.md, roadmap.md）が存在するか確認します。まだその内容は読まないでください。
- **ロードマップのチェック**: `.exec-plans/steering/roadmap.md` が存在する場合はそれを読んでください。ここには以前のディスカバリーセッションからのプロジェクトレベルのコンテキスト（アプローチ、スコープ、制約、仕様リスト）が含まれています。これを使用してプロジェクトのコンテキストを復元します。
- **最上位の構造**: プロジェクトのルートディレクトリにリストアップして、主要なディレクトリとファイルに注意を払います。サブディレクトリを再帰的に読み込まないでください。

このステップで消費するコンテキストは最小限に抑えるべきです。`specs/` が空でステアリングが存在しない場合は、「グリーンフィールドプロジェクト(新規開発)」とメモしてステップ2に進んでください。

## ステップ 2: アクションパスの決定

ステップ1からのメタデータとユーザーのリクエストに基づいて、どのパスが適用されるかを決定します:

**パス A: 既存の仕様でカバーされている**
- 要望が既存の仕様ドメイン内における拡張、改善、または修正である
- 要望のすべての意味のある部分が同仕様の境界内に収まる
- 新しい仕様を作らずとも、残りの小さなフォローアップ作業が直接対応できる
- 残りのステップをスキップする

**パス B: 仕様は不要**
- 要望がバグ修正、設定の変更、簡単なリファクタリング、または些細な追加である
- 要望の中に新しい仕様境界や既存仕様の更新を必要とする意味のある部分がない
- 既存の境界をまたがない
- 残りのステップをスキップする

**パス C: 新しい単一スコープの機能**
- リクエストが新しいものであり、既存の仕様と重複せず、1つの仕様に収まる

**パス D: 複数スコープへの分解が必要**
- リクエストが複数のドメインにまたがっている、または1つの仕様に収めると20件以上のタスクが発生する規模である

**パス E: 複合的な分解（Mixed decomposition）**
- リクエストに次のいくつかが混在している: 既存仕様の拡張、1つ以上の新しい仕様の候補、およびオプションとしての直接実装作業
- このパスは、少なくとも1つの全く新しい仕様境界が必要な場合にのみ使用する

パス C / D / E の場合、決定したパス（または複合分解）をユーザーに提示し、続行する前に確認を得てください。
パス A / B の場合、次のアクションを推奨して停止してください。

## ステップ 3: 深いコンテキストのロード

**パス C, D, E の場合にのみ実行します。** ここでディスカバリーに必要なコンテキストをロードします。

**メインコンテキストにおいて** (ユーザーとの対話に不可欠):
- **ステアリングドキュメント**: プロジェクトの目標、制約、技術スタックについて `product.md` と `tech.md` (存在する場合) を読みます
- **関連する仕様**: リクエストが既存の仕様に隣接している場合、その仕様の境界を理解し重複を避けるために、その `requirements.md` を読みます

**サブエージェントへの委任** (探索をメインコンテキストから除外する):
- **コードベースの探索**: サブエージェントを生成してコードベースを探索させ、構造化された要約を返させます。(1) 技術スタックとフレームワーク、(2) ディレクトリ構造と主要なモジュール、(3) 使用されているパターンと規則、(4) ユーザーのリクエストに関連する領域、を要約するように依頼してください。サブエージェントは200行未満の調査結果を返します。
- パス D / E の場合、自然なドメイン境界、既存のモジュール分離、そしてどの領域が既存仕様の拡張に見え、どの領域が新しい境界に見えるかを特定するようサブエージェントに依頼します。
- ステップ1のトップレベルのディレクトリリストで十分な小さな/明白なリクエストについては、サブエージェントのディスパッチをスキップしてください。

**コンテキストの予算**: メインコンテキストにロードされるコンテンツの合計は約500行未満に維持してください。重量級の探索はサブエージェントが処理します。

## ステップ 4: アイデアの理解

一度にすべてを質問するのではなく、**順次**質問し、機能の細部よりも境界の発見を優先してください:

1. **誰がなぜ**: 誰がこの問題を抱えていますか？それがどのような痛み(ペイン)/苦労を引き起こしていますか？
2. **望まれる結果**: これが完了したときに何が真(満たされている状態)であるべきですか？
3. **境界の候補 (Boundary candidates)**: この作業における自然な責任の継ぎ目はどこですか？ 実装が独立して進められるように、どこで分割できますか？
4. **境界外 (Out of boundary)**: 関連していたとしても、この仕様が明示的に所有「しない」ものは何ですか？
5. **既存との比較 (Existing vs new)**: どの部分が既存の仕様の拡張のように見え、どの部分が全く新しい境界のように見えますか？
6. **上流 / 下流**: この機能はどの既存システム、仕様、またはコンポーネントに依存していますか？ 将来の作業でこの機能に依存する可能性が高いものは何ですか？
7. **制約**: テクノロジー、スケジュール、または互換性に関する制約はありますか？

すでにロードされたコンテキストから推測できない質問のみを行ってください。ステアリングドキュメントがすでに回答している質問はスキップしてください。ユーザーがすでに明確な説明を提供している場合は、ステップ5にスキップしてください。
ここでの目標は、まだ最終的な担当を割り当てること「ではない」ことに注意してください。目標は、後で仕様、タスク、レビュースコープになるクリーンな責任の境界を発見することです。

## ステップ 5: アプローチの提案

トレードオフを伴う **2〜3 個の具体的なアプローチ**を提案します:

各アプローチについて:
- **アプローチ名**: 1行の要約
- **仕組み**: 技術的アプローチについて2〜3文
- **メリット**: このアプローチの何が良いか
- **デメリット**: リスクや欠点
- **規模の推測**: 大まかな複雑さ (小 / 中 / 大)

技術的な調査（馴染みのないフレームワーク、ライブラリの評価）が必要な場合は、サブエージェントを生成して調査させ、簡潔な要約を返させます。オプションの比較、最新バージョンの確認、既知の問題がないかのメモを依頼します。生の検索結果がメインコンテキストに入ることは絶対にありません。

1つのアプローチを推奨し、その理由を説明します。

**ユーザーがアプローチを選択した後**、ステップ6に進む前に、サブエージェントを生成して実現可能性(viability)を確認します。以下を確認するように依頼します:(1) これらの技術は現在も積極的にメンテナンスされていますか？ (2) ライセンスの非互換性（例: GPLの汚染）はありませんか？ (3) これらのコンポーネントは実際にこのユースケースで連携して動作しますか？ (4) 既知の致命的な問題（ショー・ストッパー）（重大なバグ、セキュリティ脆弱性、プラットフォームの制限）はありませんか？ 問題が見つかった場合のみ、またはすべて問題ない場合は「問題は見つかりませんでした(No issues found)」と返します。

実現可能性のチェックで問題が明らかになった場合、それをユーザーに提示し、アプローチの選択を見直します。問題がない場合は、ステップ6に進みます。

## ステップ 6: 洗練と確認

- 提案したアプローチに対するユーザーの質問や懸念に対処します
- 必要に応じてスコープを絞り込みます: より小さく、納品可能なインクリメント（漸進的開発）とクリーンな責任の継ぎ目を優先します
- パス D / E の場合: 依存関係の順序付けを伴う作業の分解(decomposition)を提案します
  - 境界を新規に設ける価値がある機能ごとに = 1つの仕様とします
  - 既存仕様の拡張は、ターゲット仕様とともに明示的にリストアップします
  - 仕様を作成するまでもない本当に小さな直接実装のタスクは、仕様に無理やり押し込まずに個別にリストアップします
  - 仕様/ワークストリーム間の依存関係を明示します
  - プロジェクトのニーズに基づいて、垂直スライス（各層を横断したエンドツーエンドでの価値）と水平レイヤー（一度に1レイヤーずつ開発）のどちらが良いかを検討します
- 最終的な方向性を確認します

## ステップ 7: デスクへのファイルの書き込み

**重要: 次のコマンドを提案する「前」に、必ずこれらのファイルをディスク(ローカル環境)に書き込まなくてはいけません。会話のテキストはセッションの境界（次回の起動時など）を越えて存続しません。このステップをスキップすると、セッションの終了時にすべてのディスカバリー分析が失われます。**

**パス C (単一の仕様) の場合**:

以下の構造で `.exec-plans/specs/<feature-name>/brief.md` をディスクに書き込みます:

```
# Brief: <feature-name>

## Problem
[誰が問題を抱えているか、それがどのような痛みを引き起こしているか]

## Current State
[今日の現状はどうか、ギャップは何か]

## Desired Outcome
[完了時に何が真であるべきか]

## Approach
[選択されたアプローチとその理由]

## Scope
- **In**: [この機能に含まれるもの]
- **Out**: [明示的に除外されるもの]

## Boundary Candidates (境界の候補)
- [責任の継ぎ目 1]
- [責任の継ぎ目 2]

## Out of Boundary (境界外)
- [この仕様が所有しない明示的な非目標(non-goals)]

## Upstream / Downstream (上流 / 下流)
- **Upstream (上流)**: [これに依存する既存システム/仕様]
- **Downstream (下流)**: [これを利用するシステム、あるいはこれに続く想定される仕様]

## Existing Spec Touchpoints (既存仕様との接点)
- **Extends (拡張)**: [この作業が更新する既存の仕様。もしあれば]
- **Adjacent (隣接)**: [重複を避けるべき隣接する仕様またはモジュール]

## Constraints (制約)
[技術的、互換性、またはその他の制約]
```

**パス D (複数仕様への分解) の場合**:

以下をディスクに書き込みます:
- `.exec-plans/steering/roadmap.md`
- `## Specs (dependency order)` の下にリストされたすべての機能のための `.exec-plans/specs/<feature>/brief.md`

以下のロードマップ構造を使用します:

```
# Roadmap

## Overview (概要)
[プロジェクトの目標と選択したアプローチ -- 1-2段落]

## Approach Decision (アプローチの決定)
- **Chosen (選択)**: [アプローチ名と要約]
- **Why (理由)**: [主要な理由]
- **Rejected alternatives (却下された代替案)**: [何が検討され、なぜ却下されたか]

## Scope (スコープ)
- **In**: [プロジェクト全体に含まれるもの]
- **Out**: [明示的に除外されるもの]

## Constraints (制約)
[技術、互換性、タイムライン、またはその他のプロジェクト全体の制約]

## Boundary Strategy (境界の戦略)
- **Why this split (なぜこの分割か)**: [これらの仕様の境界がどのように独立性を向上させるか]
- **Shared seams to watch (注意すべき共有の継ぎ目)**: [注意深いレビューを必要とする仕様間の境界]

## Specs (dependency order)
- [ ] feature-a -- [一行での説明]. Dependencies: none
- [ ] feature-b -- [一行での説明]. Dependencies: feature-a
- [ ] feature-c -- [一行での説明]. Dependencies: feature-a, feature-b
```

その後、`## Specs (dependency order)` の下にリストアップされた **「すべて」** の機能について、パスCの brief フォーマットを用いて `.exec-plans/specs/<feature>/brief.md` を書き込んでください。これにより `$tp-spec-batch` を使用した並行での仕様作成が可能になります。

**パス E (複合的な分解) の場合**:

パス D と同じロードマップの構造に加えて、以下の追加セクションを使用します:

```
## Existing Spec Updates
- [ ] existing-feature-a -- [行われる拡張の一行での説明]. Dependencies: none
- [ ] existing-feature-b -- [行われる拡張の一行での説明]. Dependencies: feature-a

## Direct Implementation Candidates
- [ ] small-item-a -- [なぜこれが直接実装にとどまるのか(別に仕様を要さないのか)]
- [ ] small-item-b -- [なぜこれが直接実装にとどまるのか(別に仕様を要さないのか)]

## Specs (dependency order)
- [ ] new-feature-a -- [一行での説明]. Dependencies: none
- [ ] new-feature-b -- [一行での説明]. Dependencies: new-feature-a
```

パス E のルール:
- `## Specs (dependency order)` は `$tp-spec-batch` が変更なしでパースできるように **新しい仕様のためだけに** 予約しておきます。
- 既存の仕様の拡張は `## Existing Spec Updates` の下に記録します
- 真の(仕様なしでの)作業(no-spec work)は `## Direct Implementation Candidates` の下に記録します。
- `brief.md` の書き出しは、`## Specs (dependency order)` の下にリストアップされた **新しい仕様(new specs)** のみに対して行います。

**再突入・再エントリー (roadmap.md が既に存在する場合)**:
次の新しい仕様のための brief.md をディスクに書き込みます。実装の経験に基づいてスコープや順序が変更された場合は roadmap.md を更新しますが、完了した項目や以前のフェーズは保持します。

書き込んだ後、ファイルを読み直してそれらが存在することを検証します。

## ステップ 8: 次のステップの提案

次のコマンドを提案して停止します。このスキルから下流の仕様生成を自動的に実行 「しないで」ください。

- パス A: 既存の仕様を更新するための `$tp-spec-requirements {feature}`
- パス B: 仕様を作成せずに直接実装することを推奨
- パス C: デフォルトは `$tp-spec-init <feature-name>`
  - オプションのファストパス: ユーザーがすぐに継続することを明示的に望む場合は `$tp-spec-quick <feature-name>`
- パス D: デフォルトは `$tp-spec-batch` (roadmap.md の依存関係の順序に基づいてすべての仕様を並行して作成します)
  - オプションの慎重なパス: ユーザーが残りのバッチ処理を行う前に最初の一部(slice)を検証したい場合は `$tp-spec-init <first-feature-name>`
- パス E: 新しい仕様部分の分解に基づいて次のコマンドを選択します
  - 新しい仕様が正確に1つである場合: `$tp-spec-init <new-feature-name>`
  - 複数の新しい仕様がある場合: `$tp-spec-batch`
  - どの既存の仕様が `$tp-spec-requirements <feature>` によって再訪問されるべきかにも言及する
- 再エントリ: `$tp-spec-init <next-feature-name>` または、複数の仕様が残っている場合は `$tp-spec-batch`

分解（decomposition）に既存仕様の更新と直接の実装候補しか含まれていない場合は、パスEを使用「しないで」ください。 明確な受け皿となる既存の仕様が1つある場合はパスAを優先するか、ロードマップへエントリーを作成せずに既存の仕様の更新と直接の実装作業をただ推奨してください。

</instructions>

## 重要な制約事項
- **ディスク上のファイルは継続性の源泉です**: パス C/D/E の場合、次のコマンドを提案する前に、必要に応じて brief.md と roadmap.md をディスクに書き込んでください。ディスカバリーの結果を会話テキストの中だけに残さないでください。

## 安全性とフォールバック

**ロードマップが既に存在している (再エントリ)**:
- 質問をする前に、roadmap.md を読み、プロジェクトのコンテキストを復元してください。
- （これまでに）完了した仕様の状態に基づいて、次の仕様を決定してください。
- 次の仕様のためだけの brief.md を(Just-in-Timeで)書き込んでください。
- もし実装での経験に基づいてスコープや順序が変更された場合は roadmap.md を更新してください。
- リクエストによってプロジェクトが拡大する場合は、新しい仕様を新たなフェーズとして追加してください。既存のコンテンツを上書きしてはいけません。
