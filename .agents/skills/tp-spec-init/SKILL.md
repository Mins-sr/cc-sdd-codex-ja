# 仕様の初期化

<instructions>
## コアタスク
プロジェクトの説明 (`$ARGUMENTS`) から一意の機能名を生成し、仕様の構造を初期化します。

## 実行手順
1. **ブリーフの確認**: `.exec-plans/specs/{feature-name}/brief.md` が存在する場合（`$tp-discovery` によって作成された場合）、それを読み込みます。ブリーフには、ディスカバリーセッションからの問題点、アプローチ、スコープ、および制約が含まれています。これらを使用してプロジェクトの説明を事前入力し、ブリーフで既に回答されている明確化のための質問をスキップします。
2. **意図の明確化**: requirements.md のプロジェクトの説明には、(a) 誰が問題を抱えているか、(b) 現在の状況、(c) 何を変更すべきか、の3つの要素が含まれている必要があります。brief.md が存在し、これらをカバーしている場合はステップ3に進みます。そうでない場合は、続行する前にユーザーに明確化を求めてください。必要なだけ質問をしてください。独自の推測でギャップを埋めないでください。
3. **一意性の検証**: `.exec-plans/specs/` で名前の競合を確認します。ディレクトリがすでに `brief.md` のみ（`spec.json` はなし）で存在する場合は、そのディレクトリを使用します（ディスカバリーによって作成されています）。
4. **ディレクトリの作成**: `.exec-plans/specs/[feature-name]/` (ディスカバリーによってすでに存在する場合はスキップします)
5. **テンプレートを使用したファイルの初期化**:
   - `.exec-plans/settings/templates/specs/init.json` を読み込む
   - `.exec-plans/settings/templates/specs/requirements-init.md` を読み込む
   - プレースホルダーを置き換える:
     - `{{FEATURE_NAME}}` → 生成された機能名
     - `{{TIMESTAMP}}` → 現在の ISO 8601 タイムスタンプ
     - `{{PROJECT_DESCRIPTION}}` → 利用可能な場合は brief.md から、それ以外の場合は `$ARGUMENTS` から
     - `ja` → 言語コード（ユーザーの入力言語から検出、デフォルトは `en`）
   - `spec.json` と `requirements.md` を仕様ディレクトリに書き込む

## 重要な制約
- 要件、設計、またはタスクを生成してはいけません。このスキルは spec.json と requirements.md のみを作成します。
</instructions>

## 出力の説明
`spec.json` で指定された言語で、以下の構造を使用して出力を提供してください：

1. **生成された機能名**: 1〜2文の根拠を伴う `feature-name` 形式
2. **プロジェクトの概要**: 簡潔な概要（1文）
3. **作成されたファイル**: フルパスを含む箇条書きリスト
4. **次のステップ**: `$tp-spec-requirements <feature-name>` を示すコマンドブロック

**フォーマットの要件**:
- Markdown の見出し (##, ###) を使用する
- コマンドはコードブロックで囲む
- 全体の出力を簡潔に保つ（250文字以下）
- `spec.json.language` に従い、明確で専門的な言語を使用する

## 安全性とフォールバック
- **曖昧な機能名**: 機能名の生成が不明確な場合、2〜3のオプションを提案し、ユーザーに選択を求める
- **テンプレートの欠落**: テンプレートファイルが `.exec-plans/settings/templates/specs/` に存在しない場合、特定の欠落しているファイルパスを添えてエラーを報告し、リポジトリのセットアップの確認を提案する
- **ディレクトリの競合**: 機能名がすでに存在する場合、数値のサフィックス（例: `feature-name-2`）を追加し、自動的な競合解決をユーザーに通知する
- **書き込みの失敗**: 特定のパスとともにエラーを報告し、権限またはディスク容量の確認を提案する
